## Глава 33. Реактивные потоки (Flow)

### 1. Что такое Холодный стрим (Cold Stream)?

По умолчанию `Flow` является **холодным**. Это значит, что код внутри потока не начнет выполняться до тех пор, пока на него не "подпишутся" (пока кто-то не попросит выдать данные).

* **Аналогия из жизни:** Фильм на Netflix (Холодный) против Прямого эфира на ТВ (Горячий).
* Прямой эфир (Горячий поток) идет прямо сейчас. Если ты включишь телевизор на 10 минут позже, ты пропустишь начало. Телеканалу всё равно, смотрит ли его кто-то, он всё равно вещает.
* Netflix (Холодный поток `Flow`) — это просто файл на сервере. Он ничего не делает и не тратит ресурсы. Но как только ты нажимаешь кнопку "Play" (вызываешь метод `collect`), фильм начинается **специально для тебя с самой первой секунды**. Если твой друг нажмет "Play" на своем телефоне, для него запустится его собственный, независимый показ.



```kotlin
import kotlinx.coroutines.*
import kotlinx.coroutines.flow.*

// Создаем "чертеж" потока. Слово suspend здесь не нужно!
val coldFlow: Flow<Int> = flow {
    println("▶️ Поток начал работу!")
    for (i in 1..3) {
        delay(1000) // Мы можем использовать suspend-функции внутри!
        emit(i)     // emit - "выплюнуть" (отправить) значение наружу
    }
}

fun main() = runBlocking {
    println("Создали coldFlow. Ждем...")
    delay(2000) // Прошло 2 секунды, но flow внутри даже не дернулся.

    println("Начинаем сбор (collect) #1:")
    // collect - это терминальная (конечная) операция. 
    // Именно она "нажимает Play" и запускает генерацию.
    coldFlow.collect { number -> 
        println("Получили: $number")
    }

    println("Начинаем сбор (collect) #2:")
    // Второй вызов collect запустит код внутри flow с самого начала!
    coldFlow.collect { number -> 
        println("Снова получили: $number")
    }
}

```

### 2. Трансформация (Асинхронный конвейер)

Так же, как и обычные коллекции, `Flow` можно фильтровать (`filter`), преобразовывать (`map`) и объединять. Но здесь есть суперсила: **каждый оператор трансформации может вызывать `suspend` функции**.

* **Аналогия из жизни:** Заводской конвейер сборки автомобилей. Кузов едет по ленте (`Flow`). Рабочий номер один (`map`) красит кузов. Если краска сохнет долго, рабочий может спокойно "уснуть" (`delay`), не останавливая весь завод, а потом продолжить работу. Затем кузов едет к инспектору (`filter`), который тоже может взять паузу на проверку в базе данных (`suspend`), прежде чем пропустить машину дальше.

```kotlin
suspend fun fetchDiscount(productId: Int): Int {
    delay(500) // Имитация запроса к серверу скидок
    return if (productId % 2 == 0) 20 else 0
}

fun getProductsFlow() = flow {
    emit(1) // ID товара
    emit(2)
}

fun main() = runBlocking {
    getProductsFlow()
        .filter { id -> 
            println("Фильтруем $id...")
            id > 0 
        }
        .map { id -> 
            // МАГИЯ: Мы вызываем suspend-функцию прямо внутри map!
            // В обычных списках listOf().map {} так сделать нельзя.
            val discount = fetchDiscount(id) 
            "Товар $id (Скидка $discount%)"
        }
        .collect { result -> 
            println("Готово: $result")
        }
}

```

### 3. Управление контекстом (flowOn) и Защита от дурака

В прошлом модуле (6.2) мы изучили `withContext` для переключения пулов потоков (например, прыжок в `Dispatchers.IO` для работы с сетью).

**Строгое правило Flow: Использовать `withContext` внутри блока `flow { ... }` КАТЕГОРИЧЕСКИ ЗАПРЕЩЕНО.** Программа просто упадет с ошибкой `IllegalStateException`.

Почему? Разработчики Kotlin внедрили принцип **Сохранения контекста (Context Preservation)**. Человек, который вызывает `.collect {}` в главном потоке (UI), должен быть уверен, что никто внутри `Flow` не подменит ему поток и не сломает интерфейс.

Чтобы безопасно поменять диспетчер для тяжелых операций (например, чтения из базы), был создан оператор **`flowOn`**.

* **Аналогия из жизни:** Водопровод (Upstream и Downstream).
Ты стоишь в квартире на первом этаже и открываешь кран (вызываешь `collect` в `Dispatchers.Main`). Вода течет к тебе.
Где-то далеко в горах насос качает эту воду (вызывает `emit`). Если насосу нужна промышленная электросеть (`Dispatchers.IO`), он не должен тянуть её в твою квартиру. Он ставит трансформатор (`flowOn`) **выше по течению**. Всё, что выше трансформатора, работает на промышленной сети (IO). Всё, что ниже — течет в твою квартиру на обычной (Main).

```kotlin
// Функция возвращает Flow. Мы не знаем, на каком потоке её вызовут.
fun downloadFileFlow(): Flow<String> = flow {
    // Этот код выполнится в пуле Dispatchers.IO благодаря flowOn ниже!
    println("[${Thread.currentThread().name}] Читаем жесткий диск...")
    emit("Кусок файла 1")
    delay(100)
    emit("Кусок файла 2")
}
.flowOn(Dispatchers.IO) // Меняет контекст ТОЛЬКО для кода ВЫШЕ (upstream)

fun main() = runBlocking(Dispatchers.Main) { // Представим, что это главный поток UI
    
    downloadFileFlow()
        .map { text -> 
            // Этот код выполнится в Dispatchers.Main, 
            // потому что flowOn остался ВЫШЕ по течению!
            println("[${Thread.currentThread().name}] Трансформируем...")
            text.uppercase() 
        }
        .collect { result ->
            // Это тоже выполнится в Dispatchers.Main
            println("[${Thread.currentThread().name}] Отрисовываем на экране: $result")
        }
}

```

**Как это будет в логах:**

1. `[DefaultDispatcher-worker-1]` Читаем жесткий диск... *(Это IO пул!)*
2. `[main]` Трансформируем...
3. `[main]` Отрисовываем на экране: КУСОК ФАЙЛА 1

Оператор `flowOn` элегантно разделяет обязанности: генерация данных может происходить в тяжелых фоновых потоках, а их обработка и потребление — в легком главном потоке, и они никогда не перепутаются.

---

Итог:

* **`Flow`** — это асинхронный аналог `Sequence`. Он позволяет поштучно генерировать и обрабатывать потоки данных, используя `suspend` функции на каждом этапе (генерация, фильтрация, трансформация).
* `Flow` является **холодным стримом**: код внутри билдера `flow { ... }` не выполняется при создании объекта. Он запускается только в момент вызова терминального оператора (например, `collect`), и для каждого нового "коллектора" код выполняется заново.
* **Context Preservation** — фундаментальное свойство `Flow`. Внутри `flow { emit() }` нельзя использовать `withContext(Dispatcher)`, чтобы сменить поток, так как это нарушит прозрачность для подписчика.
* Для смены пула потоков (Диспетчера) при генерации данных используется оператор **`flowOn(Dispatcher)`**. Он меняет контекст исключительно для операций **выше по течению (upstream)**, оставляя все операции **ниже по течению (downstream)**, включая сам `collect`, в оригинальном потоке вызывающей стороны.
