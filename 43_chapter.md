## Глава 43. Кодогенерация

### 1. Проблема: Костыльная архитектура KAPT

**KAPT (Kotlin Annotation Processing Tool)** был создан от безысходности. Исторически все процессоры аннотаций (Dagger, Glide, Room) были написаны для языка Java. Они использовали встроенный в Java компилятор (`javac`) и его API для чтения кода.

Когда появился Kotlin, возникла проблема: Java-компилятор `javac` **не понимает код на Kotlin**. Он смотрит на `val`, `data class` и `?` и выдает ошибку.

Чтобы старые Java-библиотеки заработали в Kotlin-проектах, разработчикам пришлось создать костыль — KAPT.

* **Аналогия из жизни:** Представь, что ты написал гениальную книгу на русском языке (Kotlin). Ты хочешь отдать её на вычитку знаменитому редактору (Процессору аннотаций Dagger). Но редактор знает **только английский** (Java).
Что делает KAPT? Он нанимает переводчика, который берет твою книгу и переводит её на ломаный английский. Этот перевод называется **Java Stubs (Стабы)**. Редактор читает английский текст, пишет свои заметки, а потом ты продолжаешь работу.

**Почему KAPT — это боль?**

1. **Генерация стабов (Stubs Generation):** Это невероятно медленно. Прежде чем процессор аннотаций вообще начнет работать, компилятор Kotlin должен пробежаться по всему твоему коду и сгенерировать фейковые Java-классы с пустыми методами. Этот этап съедает до 30-40% всего времени сборки проекта!
2. **Потеря контекста Kotlin:** Поскольку KAPT переводит код в Java, процессор аннотаций "слепнет". В Java нет Nullable-типов. Kotlin `String?` превращается в обычный Java `String` с аннотацией `@Nullable`. `data class` теряет свою магию и выглядит как обычный класс с геттерами. Процессор не видит специфичных фич Kotlin (например, sealed-классов или extension-функций) в их первозданном виде.

### 2. Решение: Родная архитектура KSP

**KSP (Kotlin Symbol Processing)** — это API от Google и JetBrains, написанное с нуля специально для Kotlin.

Его главная идея: **Выкинуть Java из уравнения.**

* **Аналогия из жизни:** Ты больше не отправляешь книгу англоязычному редактору. Ты нанимаешь **русскоязычного редактора** (KSP-плагин), который читает твой текст в оригинале, понимая все метафоры, сленг и игру слов (фичи языка Kotlin).

**Как это работает:**
KSP встраивается напрямую во фронтенд компилятора Kotlin. Он не генерирует никаких Java-стабов. Он просто смотрит на Abstract Syntax Tree (Абстрактное синтаксическое дерево) твоего Kotlin-кода и напрямую анализирует **Символы** (классы, свойства, функции).

**Преимущества KSP:**

1. **Скорость:** Из-за отсутствия генерации Java-стабов, KSP работает **до 2 раз быстрее** чем KAPT. Для больших проектов это экономит минуты при каждой сборке.
2. **Глубокое понимание Kotlin:** KSP-процессор изначально знает, что такое `typealias`, `inline class` (value class), корутины (`suspend`), и строгая null-безопасность. Авторам библиотек (например, Room) стало гораздо проще генерировать безопасный и оптимизированный Kotlin-код.
3. **Кроссплатформенность (KMP):** KAPT был жестко привязан к JVM (виртуальной машине Java). KSP не зависит от Java. Он работает на уровне самого Kotlin, а значит, кодогенерация теперь доступна для Kotlin Native (iOS) и Kotlin JS!

### 3. Переход в реальном мире

Сегодня вся Android/Kotlin экосистема находится в стадии активного перехода.

* База данных **Room** уже полностью поддерживает KSP и генерирует код на чистом Kotlin (раньше генерировался Java-код).
* Библиотека сериализации **Moshi** использует KSP (или свой плагин `kotlinx.serialization`, который тоже не использует KAPT).
* **Dagger / Hilt** (самый тяжелый и медленный инструмент в проекте) завершил внедрение KSP, и миграция на него радикально ускоряет сборку Enterprise-проектов.

Если в твоем `build.gradle` всё ещё есть плагин `kotlin-kapt` и зависимости `kapt(...)` — это первый кандидат на рефакторинг. Замена их на `com.google.devtools.ksp` и `ksp(...)` — это самый дешевый способ ускорить работу всей команды разработчиков.

---

Итог:

* **Кодогенерация** — механизм автоматического написания шаблонного кода (Boilerplate) на основе аннотаций, активно применяемый в DI (Dagger), базах данных (Room) и сетевых слоях (Retrofit).
* **KAPT (Kotlin Annotation Processing Tool)** — устаревший подход. Из-за того, что старые процессоры работают только с Java-кодом, KAPT вынужден предварительно генерировать промежуточные "Java-стабы" (Java Stubs) из Kotlin-кода. Это колоссально замедляет сборку и приводит к потере строгой информации о Kotlin-типах (например, Nullability).
* **KSP (Kotlin Symbol Processing)** — современный инструмент, интегрированный напрямую в компилятор Kotlin.
* **Преимущества KSP:** Отказ от генерации Java-стабов увеличивает скорость сборки до 2 раз. KSP напрямую понимает специфичные конструкции языка Kotlin (корутины, sealed-классы, value-классы) и не зависит от JVM, что позволяет использовать его в мультиплатформенных проектах (KMP).
