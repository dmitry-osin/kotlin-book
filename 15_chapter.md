## Глава 15. Коллекции и Sequence

### 1. Энергичные вычисления (Коллекции)

По умолчанию стандартные списки в Kotlin (например, `List` или `Set`) являются **энергичными (eager)**. Это означает, что при вызове любого метода трансформации (например, фильтрации) операция применяется сразу ко **всем** элементам списка, и создается **новый промежуточный список** в памяти.

* **Аналогия из жизни:** Бригада маляров красит партию из 1000 табуреток. Энергичный подход: они сначала шкурят *все 1000* табуреток (складывая их в отдельную кучу). Затем они грунтуют *все 1000* отшкуренных табуреток (создавая третью кучу). И только потом красят. Для каждой стадии требуется огромный склад (оперативная память) для хранения промежуточных результатов.

```kotlin
fun main() {
    val numbers = listOf(1, 2, 3, 4, 5)

    // Энергичный подход
    val result = numbers
        .map { 
            println("Умножаю $it")
            it * 2 
        } // Здесь создастся НОВЫЙ список в памяти: [2, 4, 6, 8, 10]
        .filter { 
            println("Фильтрую $it")
            it > 5 
        } // Здесь создастся ЕЩЕ ОДИН список: [6, 8, 10]
        .first() // Берем первый подходящий элемент (6)

    println("Результат: $result")
    // В консоли мы увидим, что программа честно умножила ВСЕ 5 чисел, 
    // затем отфильтровала ВСЕ 5 результатов, хотя нам нужно было найти только ПЕРВОЕ число.
    // Это пустая трата ресурсов процессора.
}

```

### 2. Ленивые вычисления (Sequence)

Когда данных очень много (тысячи элементов) или цепочка операций длинная, создание промежуточных списков "съест" всю оперативную память. Здесь на помощь приходят **Последовательности (`Sequence`)**.

Они работают **лениво (lazy)**. Это значит, что они не создают никаких промежуточных списков. Они берут *один* элемент и протаскивают его через всю цепочку операций (сверху вниз), прежде чем перейти к следующему элементу.

* **Аналогия из жизни:** Конвейерная лента на заводе. Берется ровно *одна* табуретка. Она сразу шкурится, грунтуется и красится, после чего выдается заказчику. Если заказчику нужна была только одна готовая табуретка, конвейер мгновенно останавливается. Остальные 999 табуреток даже не достают со склада.

Чтобы включить эту магию, достаточно вызвать метод `.asSequence()` у любой коллекции.

```kotlin
fun main() {
    val numbers = listOf(1, 2, 3, 4, 5)

    // Ленивый подход через Sequence
    val result = numbers.asSequence()
        .map { 
            println("Умножаю $it")
            it * 2 
        } // Никакого списка не создается! Только записывается правило.
        .filter { 
            println("Фильтрую $it")
            it > 5 
        } // Снова просто записывается правило.
        .first() // Терминальная (конечная) операция. Она запускает конвейер!

    println("Результат: $result")
    // В консоли мы увидим совершенно другую картину:
    // Умножаю 1 -> Фильтрую 2 (не прошло)
    // Умножаю 2 -> Фильтрую 4 (не прошло)
    // Умножаю 3 -> Фильтрую 6 (ПРОШЛО!)
    // Конвейер останавливается! Числа 4 и 5 даже не будут обрабатываться. 
    // Мы сэкономили кучу времени и памяти.
}

```

*Важное правило: Sequence не начнет ничего делать, пока в конце цепочки не будет вызвана функция, требующая реального результата (например, `toList()`, `first()`, `count()`).*

### 3. Генераторы `sequence { yield() }`

Последовательности можно не только создавать из готовых списков, но и генерировать с нуля. Для этого используется блок `sequence { ... }` и специальное слово `yield()` (выдать/уступить).

Это уникальная конструкция: когда код доходит до `yield()`, функция выдает элемент наружу и **"замораживается"** (ставится на паузу), сохраняя все свои внутренние переменные. Она проснется и продолжит работу только тогда, когда кто-то снаружи попросит следующий элемент.

* **Аналогия из жизни:** Торговый автомат с газировкой. Внутри него огромная логика и сотни банок. Но когда ты нажимаешь кнопку, механизм срабатывает ровно один раз, выдает тебе одну банку (вызов `yield()`) и снова замирает. Автомат не вываливает на тебя все свои запасы разом. Он ждет следующего нажатия кнопки.

```kotlin
// Создаем бесконечный генератор четных чисел
val evenNumbersGenerator: Sequence<Int> = sequence {
    var number = 0
    // Бесконечный цикл! В обычной коллекции это привело бы к зависанию компьютера
    // и нехватке памяти (OutOfMemoryError). Но в Sequence это безопасно.
    while (true) {
        println("-> Автомат проснулся, генерирую число: $number")
        
        // Выдаем число наружу и ЗАМОРАЖИВАЕМ выполнение функции на этой строчке
        yield(number) 
        
        // Эта часть кода выполнится только когда снаружи запросят следующее число
        number += 2 
    }
}

fun main() {
    println("Начинаем работу с генератором:")
    
    // Запрашиваем из бесконечного автомата только 3 элемента (нажали кнопку 3 раза)
    // и превращаем их в обычный список
    val threeNumbers = evenNumbersGenerator.take(3).toList()
    
    println("Получили список: $threeNumbers")
}

```

Генераторы идеальны для пагинации (когда мы пошагово подгружаем данные из сети страница за страницей) или работы со сложными математическими вычислениями, где мы не знаем заранее, сколько элементов нам понадобится.

---

Итог:

* Энергичные вычисления (стандартные коллекции `List`, `Set`) применяют каждую операцию в цепочке сразу ко всем элементам, создавая промежуточные списки в памяти. Это подходит для небольших наборов данных.
* Ленивые вычисления (`Sequence`) обрабатывают элементы поштучно (протаскивая один элемент через всю цепочку). Они не создают промежуточных списков в памяти и прерывают обработку, как только цель достигнута, что критически важно для производительности на больших объемах данных.
* Чтобы превратить обычную коллекцию в ленивую, используется функция-расширение `.asSequence()`.
* Блок `sequence { yield() }` позволяет создавать собственные ленивые генераторы (в том числе бесконечные). Ключевое слово `yield()` выдает один элемент и ставит выполнение функции "на паузу" до тех пор, пока не будет запрошен следующий элемент.
