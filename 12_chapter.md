## Глава 12. Functional Interfaces (SAM)

### 1. Проблема: Бойлерплейт анонимных объектов

Представь, что у нас есть система безопасности, которая требует передать ей "Проверяющего" (интерфейс).

* **Аналогия из жизни:** Оформление разового пропуска. У тебя есть простой и понятный договор: «Я, такой-то, обязуюсь не ломать станки».
Если мы используем старый подход (анонимные объекты), процесс выглядит как бюрократический ад: тебе выдают бланк на 5 страниц, ты должен написать шапку "Я подтверждаю, что являюсь Проверяющим", поставить печать, расписаться на каждой странице и только в конце написать суть.

```kotlin
// Обычный интерфейс с одним методом
interface SecurityCheck {
    fun checkPass(id: String): Boolean
}

// Функция, которая принимает этот интерфейс
fun enterFacility(id: String, security: SecurityCheck) {
    if (security.checkPass(id)) println("Проход открыт") else println("Тревога!")
}

fun main() {
    // Бюрократия: создаем анонимный объект (object expression).
    // Мы пишем кучу лишних слов: object, :, SecurityCheck, override, fun, checkPass...
    enterFacility("ID-123", object : SecurityCheck {
        override fun checkPass(id: String): Boolean {
            return id.startsWith("ID-")
        }
    })
}

```

### 2. Решение: `fun interface` и SAM-конверсия

Если мы добавим перед словом `interface` ключевое слово **`fun`**, мы скажем компилятору: *"Это строгий контракт ровно на одно действие. Разреши программистам не заполнять бюрократические бланки, пусть они передают только саму суть"*.

* **Аналогия из жизни:** Сканер отпечатков пальцев вместо бумажного пропуска. Тебе больше не нужно заполнять бланки, называть свое имя и ставить печати. Ты просто прикладываешь палец (передаешь чистую логику — лямбду). Система сама понимает, что твой палец идеально подходит под контракт "Проверка личности" (это и есть SAM-конверсия).

```kotlin
// 1. Добавляем слово 'fun'.
// Правило: внутри должен быть строго ОДИН метод без реализации (абстрактный).
fun interface FastSecurityCheck {
    fun checkPass(id: String): Boolean
}

fun enterFastFacility(id: String, security: FastSecurityCheck) {
    if (security.checkPass(id)) println("Проход открыт") else println("Тревога!")
}

fun main() {
    // 2. SAM-конверсия в действии!
    // Мы выкидываем слова object, override, имя метода. 
    // Мы просто передаем лямбда-выражение { ... }. 
    // Компилятор Kotlin сам "оборачивает" эту лямбду в нужный интерфейс под капотом.
    enterFastFacility("ID-123") { passId -> 
        passId.startsWith("ID-") 
    }
}

```

### 3. Строгий контракт SAM

Почему это называется **SAM** (*Single Abstract Method*)? Потому что компилятор сможет автоматически превратить лямбду в объект только в том случае, если у него нет сомнений, какой именно метод ты реализуешь.

Если в интерфейсе будет два пустых метода, компилятор запутается: *"К какому из этих двух методов относится код в твоей лямбде?"*. Поэтому `fun interface` выдаст ошибку компиляции, если абстрактных методов больше одного.

При этом функциональный интерфейс может быть сложным и содержать другие методы, **если у них уже есть готовая реализация**.

```kotlin
fun interface PaymentValidator {
    // Это тот самый ЕДИНСТВЕННЫЙ абстрактный метод, который заменится лямбдой
    fun validate(amount: Double): Boolean 

    // Это обычный метод с реализацией. Он не нарушает правило SAM.
    // Таких методов может быть сколько угодно.
    fun printRules() {
        println("Сумма должна быть больше нуля и не превышать лимит.")
    }
}

fun main() {
    // Мы создаем переменную типа PaymentValidator, передавая только лямбду для validate().
    val myValidator = PaymentValidator { amount -> amount > 0 && amount <= 1000 }

    // Теперь мы можем использовать и нашу лямбду, и готовый метод из интерфейса:
    myValidator.printRules()
    println("Проверка 500: ${myValidator.validate(500.0)}")
}

```

*(Кстати, если ты работаешь с Java-библиотеками (например, Android SDK или Spring), где интерфейсы написаны на Java и имеют только один метод (например, `Runnable` или `OnClickListener`), Kotlin автоматически применяет к ним SAM-конверсию. Тебе не нужно ничего делать, ты можешь сразу передавать лямбды).*

---

Итог:

* Если интерфейс содержит только один абстрактный (нереализованный) метод, он называется **SAM-интерфейсом** (Single Abstract Method).
* В Kotlin такой интерфейс объявляется с помощью ключевых слов `fun interface`.
* Главное преимущество `fun interface` заключается в **SAM-конверсии**: она позволяет отказаться от громоздкого создания анонимных объектов (через `object : InterfaceName`) и передавать вместо них лаконичные лямбда-выражения `{ ... }`.
* Внутри `fun interface` может находиться любое количество методов с уже готовой реализацией, но метод без реализации (абстрактный) должен быть строго один, иначе компилятор не сможет сопоставить его с переданной лямбдой.
