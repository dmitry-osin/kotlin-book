## Глава 19. Создание DSL (Domain Specific Language)

### 1. Как работает DSL? (Аналогия)

В прошлом модуле (4.2) мы изучили лямбды с ресивером. Если объединить их вместе и вложить друг в друга, мы получим древовидную структуру, которая читается как конфигурационный файл.

* **Аналогия из жизни:** Заказ сэндвича в ресторане (например, в Subway).
* Обычный (императивный) код — это когда ты пытаешься управлять поваром как роботом: *"Возьми муку, испеки хлеб, отрежь кусок, положи сыр, закрой вторым куском"*. Это долго и чревато ошибками.
* DSL — это специальный бланк ресторана, где ты просто ставишь галочки в нужных секциях: *"Хлеб: Итальянский. Начинка: { Мясо: Индейка, Сыр: Чеддер }"*. Ты говоришь на языке предметной области (на языке сэндвичей), а как именно это готовить — ресторан разберется сам.



### 2. Пишем свой мини-HTML DSL

Давай создадим генератор веб-страниц. В Java для этого пришлось бы создавать объекты, вызывать методы `appendChild()` и склеивать строки. В Kotlin мы сделаем так, чтобы код выглядел почти как настоящий HTML.

Для этого нам понадобятся обычные классы и функции, принимающие лямбды с ресивером (`Тип.() -> Unit`).

```kotlin
// 1. Создаем базовые "кирпичики" нашей предметной области.
// Класс, описывающий абзац текста (тег <p>)
class Paragraph {
    var text: String = ""
    
    // Метод для превращения объекта в HTML-строку
    fun render() = "<p>$text</p>"
}

// Класс, описывающий тело страницы (тег <body>)
class Body {
    // В теле может быть много абзацев
    private val paragraphs = mutableListOf<Paragraph>()

    // Функция принимает лямбду с ресивером Paragraph.
    // Это значит, внутри лямбды мы будем настраивать конкретный абзац.
    fun p(setupBlock: Paragraph.() -> Unit) {
        val paragraph = Paragraph()
        paragraph.setupBlock() // Выполняем настройку
        paragraphs.add(paragraph) // Сохраняем абзац
    }

    fun render(): String {
        val renderedParagraphs = paragraphs.joinToString("\n") { it.render() }
        return "<body>\n$renderedParagraphs\n</body>"
    }
}

// Класс, описывающий саму страницу (тег <html>)
class HTMLPage {
    private var body: Body? = null

    // Функция принимает лямбду с ресивером Body.
    fun body(setupBlock: Body.() -> Unit) {
        val b = Body()
        b.setupBlock()
        body = b
    }

    fun render(): String = "<html>\n${body?.render() ?: ""}\n</html>"
}

// 2. Главная точка входа в наш DSL.
// Эта глобальная функция запускает весь процесс.
fun html(setupBlock: HTMLPage.() -> Unit): HTMLPage {
    val page = HTMLPage()
    page.setupBlock()
    return page
}

```

Теперь посмотрим, как выглядит использование нашего DSL. Сравни это с классическим созданием объектов — магия лямбд с ресивером убирает весь визуальный мусор:

```kotlin
fun main() {
    // Вызов функции html. Открываем скобки и мы уже внутри объекта HTMLPage!
    val myPage = html {
        
        // Внутри HTMLPage мы вызываем метод body. 
        // Открываем скобки - и мы внутри объекта Body!
        body {
            
            // Внутри Body мы вызываем метод p. 
            // Открываем скобки - и мы внутри объекта Paragraph!
            p {
                text = "Привет! Это мой первый абзац, сгенерированный через DSL."
            }
            
            p {
                text = "Код читается как обычный текст, но при этом компилятор строго следит за типами."
            }
        }
    }

    println(myPage.render())
}

```

**Результат вывода:**

```html
<html>
<body>
<p>Привет! Это мой первый абзац, сгенерированный через DSL.</p>
<p>Код читается как обычный текст, но при этом компилятор строго следит за типами.</p>
</body>
</html>

```

### 3. Защита от глупостей: Аннотация `@DslMarker`

В коде выше есть одна архитектурная проблема. Поскольку лямбды вкладываются друг в друга, внутренние блоки "видят" методы внешних блоков (это называется *неявный this*).

Это значит, что внутри абзаца `p { ... }` мы могли бы случайно снова вызвать метод `body { ... }`, потому что компилятор помнит, что мы находимся внутри `HTMLPage`. В результате получится бессмыслица.

* **Аналогия из жизни:** Зоны доступа на кухне ресторана. Если повар зашел в холодильник (блок `p`), он не должен иметь возможности прямо оттуда включить плиту (команда из блока `HTMLPage`), не выйдя предварительно из холодильника.

Чтобы ограничить видимость, в Kotlin есть специальная аннотация `@DslMarker`.

```kotlin
// 1. Создаем свою собственную аннотацию для нашего DSL
@DslMarker
annotation class HtmlDsl

// 2. Помечаем этой аннотацией все классы, которые участвуют в нашем мини-языке
@HtmlDsl
class HTMLPage { /* ... */ }

@HtmlDsl
class Body { /* ... */ }

@HtmlDsl
class Paragraph { /* ... */ }

```

Теперь компилятор создаст строгие границы. Если мы попытаемся внутри `p { ... }` вызвать метод `body { ... }`, компилятор выдаст ошибку, защищая нас от создания неправильной структуры документа.

Именно на таких Type-Safe Builders в Kotlin строятся сверхмощные инструменты: генерация SQL-запросов (библиотека *Exposed*), верстка интерфейсов в Android (*Jetpack Compose*) и написание скриптов сборки (*Gradle Kotlin DSL*).

---

Итог:

* DSL (Domain Specific Language) — это специализированный мини-язык для решения конкретной узкой задачи, который выглядит как конфигурационный файл, но при этом является строгим, компилируемым кодом.
* В основе создания DSL в Kotlin лежат типобезопасные строители (Type-Safe Builders), которые создаются путем комбинации функций высшего порядка и лямбд с ресивером (`Type.() -> Unit`).
* Благодаря лямбдам с ресивером мы можем вкладывать блоки кода друг в друга (создавая древовидную структуру, например, HTML-тегов или JSON-объектов) без необходимости каждый раз писать имя родительского объекта.
* Для предотвращения логических ошибок (например, вызова метода верхнего уровня внутри вложенного блока) используется аннотация `@DslMarker`, которая ограничивает область видимости неявных приемников (`this`), строго изолируя "уровни" нашего мини-языка друг от друга.
