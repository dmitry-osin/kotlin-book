## Глава 31. Structured Concurrency и Кооперативная многозадачность

### 1. Структурированная конкурентность и Иерархия (Job)

В Kotlin корутины живут не сами по себе, а в строгой иерархии (Дереве). У каждой корутины есть свой объект **`Job`** (Задача). Когда ты запускаешь новую корутину внутри другой, создается связь «Родитель-Ребенок».

Из этой иерархии вытекают два железных правила:

1. **Родитель не может завершиться, пока не завершат работу все его дети.** Он будет терпеливо ждать их выполнения.
2. **Каскадная отмена:** Если ты отменяешь родителя (например, пользователь ушел с экрана), родитель немедленно отменяет всех своих детей, внуков и правнуков. Никаких утечек памяти.

* **Аналогия из жизни:** Строительная компания. Есть Главный Архитектор (Родительская корутина). Он нанимает Бригадира и Электрика (Дочерние корутины).
* *Правило 1:* Архитектор не может сдать проект заказчику, пока Бригадир и Электрик не закончат работу.
* *Правило 2:* Если заказчик звонит и говорит "Отменяем стройку!", Архитектору не нужно бегать за каждым рабочим. Он просто дает команду по рации, и все немедленно бросают инструменты.



```kotlin
import kotlinx.coroutines.*

suspend fun buildHouse() = coroutineScope { // Создаем область (scope) для корутин
    println("Архитектор: Начинаем стройку!")

    // Запускаем первого ребенка
    val builderJob = launch {
        delay(1000) // Имитация долгой работы
        println("Бригадир: Я построил стены")
    }

    // Запускаем второго ребенка
    val electricianJob = launch {
        delay(3000)
        println("Электрик: Я провел свет")
    }

    delay(500)
    println("Архитектор: Стройка отменяется! У нас кончились деньги.")
    
    // Отменяем ВСЮ область (самого Архитектора). 
    // Это автоматически убьет и builderJob, и electricianJob!
    cancel() 
}
// В консоли мы НЕ УВИДИМ сообщений от Бригадира и Электрика. 
// Они были отменены до того, как успели доделать работу.

```

### 2. Ошибка одного убивает всех: Проблема обычного `Job`

У классической иерархии `Job` есть одна опасная особенность. Если дочерняя корутина не просто отменяется, а **падает с ошибкой (Exception)** (например, пропал интернет), она начинает паниковать.
Она кричит родителю: *"Я упала!"*. Родитель отменяет ВСЕХ остальных детей и падает сам.

* **Аналогия из жизни:** Синхронное плавание. Если у одной пловчихи свело ногу (ошибка), выступление останавливается для всей команды.

Но в бизнесе так бывает не всегда. Если мы грузим ленту новостей: картинки, текст и аватарку. Если аватарка не скачалась (ошибка), мы не хотим удалять текст и картинки.

#### Решение: `SupervisorJob`

Для таких случаев придумали **`SupervisorJob`** (или `supervisorScope`). Он меняет правило поведения при ошибках: *"Дети независимы. Если один ребенок упал с ошибкой, родитель игнорирует это и позволяет остальным детям продолжить работу"*.

* **Аналогия из жизни:** Колл-центр. Если у одного оператора оборвалась связь с клиентом (ошибка), остальные 50 операторов продолжают спокойно разговаривать по своим телефонам. Начальник (Supervisor) просто помечает проблему в журнал.

```kotlin
suspend fun loadDashboard() = supervisorScope { // Используем Supervisor!
    
    val textJob = launch {
        delay(1000)
        println("Текст успешно загружен")
    }

    val avatarJob = launch {
        delay(500)
        throw Exception("ОШИБКА 404: Аватарка не найдена!") // Падаем с ошибкой!
    }
}
// Результат: 
// Несмотря на то, что avatarJob взорвался на 500-й миллисекунде, 
// textJob спокойно доработает и напечатает свое сообщение!
// Если бы мы использовали обычный coroutineScope, textJob был бы убит.

```

### 3. Кооперативная отмена и магия `yield()`

А теперь самое интересное на уровне железа. В Kotlin нельзя просто взять и "убить" корутину снаружи (как это было со старыми потоками `Thread.stop()`). Отмена корутин работает по принципу **Кооперативности**.

Это значит, что родитель может попросить корутину остановиться (`job.cancel()`), но если корутина занята тяжелой безостановочной математикой и не проверяет свою "почту", она проигнорирует отмену и продолжит сжигать процессор!

* **Аналогия из жизни:** Дебаты политиков. Обычная корутина, занятая математикой — это спикер, который смотрит в пол и тараторит текст без остановок. Модератор (Диспетчер) машет красным флагом "Твое время вышло!" (вызван `cancel()`), но спикер этого не видит. Он блокирует трибуну.

**Как сделать корутину вежливой?**
Она должна периодически проверять, не отменили ли её. Все стандартные `suspend` функции (типа `delay()`) уже умеют это делать. Но если у тебя сплошная математика (цикл `while`), тебе нужно делать это руками.

У нас есть два пути:

1. **Свойство `isActive` (или метод `ensureActive()`):** Это взгляд на модератора. *"Я еще активен? Да? Продолжаю считать"*.
2. **Функция `yield()`:** Это высшая степень вежливости.

#### Функция `yield()` (Уступить)

`yield()` — это специальная `suspend` функция. Она делает две вещи:

1. Она проверяет, не отменили ли корутину. Если отменили — немедленно выбрасывает `CancellationException` и прекращает работу.
2. Она **добровольно отдает поток (Thread)** обратно Диспетчеру, говоря: *"Я немного поработал. Если в очереди есть другие корутины, пусть они тоже воспользуются процессором, а я встану в конец очереди"*.

* **Аналогия из жизни:** Вежливый спикер на дебатах. Он говорит два предложения, затем замолкает на секунду (`yield()`), смотрит на модератора (не показывают ли красный флаг?) и дает возможность другим спикерам вставить слово. Если флага нет, он продолжает свою речь.

```kotlin
fun main() = runBlocking {
    val heavyJob = launch(Dispatchers.Default) {
        var count = 0L
        
        // Долгий цикл
        while (count < 100_000_000L) {
            count++
            
            // Если мы не напишем yield(), корутина захватит поток 
            // и проигнорирует отмену внизу!
            
            if (count % 1_000_000L == 0L) {
                // Раз в миллион итераций мы "поднимаем голову" и уступаем трибуну
                yield() 
                println("Посчитал до $count, продолжаю...")
            }
        }
    }

    delay(10) // Даем корутине поработать 10 миллисекунд
    println("Модератор: Ваше время вышло, отмена!")
    heavyJob.cancel() // Просим отмениться
}
// Благодаря yield() корутина мгновенно прервется, как только дойдет 
// до следующей проверки, не загружая процессор впустую.

```

---

Итог:

* **Структурированная конкурентность** гарантирует, что корутины не "утекают". Запуск происходит в определенных областях видимости (scopes), где родители следят за жизненным циклом своих детей через объекты `Job`.
* Отмена родителя каскадно отменяет всех потомков.
* Обычный `Job` ведет себя строго: ошибка одного дочернего процесса отменяет всех остальных. **`SupervisorJob`** делает дочерние процессы независимыми — ошибка одного не влияет на "братьев" и родителя.
* **Отмена в Kotlin — кооперативная**. Корутина не может быть прервана извне в момент выполнения тяжелых синхронных вычислений (CPU-bound). Разработчик обязан сам проверять статус отмены.
* Функция **`yield()`** — это `suspend` функция, решающая сразу две проблемы тяжелых циклов: она проверяет статус отмены корутины и "вежливо" освобождает текущий поток Диспетчера, позволяя другим корутинам выполнить свою часть работы, тем самым предотвращая блокировку пула потоков.
