## Глава 27. Рефлексия (kotlin-reflect)

### 1. Отличие `KClass` (Kotlin) от `Class` (Java)

Так как Kotlin работает поверх виртуальной машины Java (JVM), он унаследовал старую систему рефлексии Java — класс `java.lang.Class`. Но мы уже знаем, что Kotlin гораздо богаче: в нем есть свойства (вместо полей), Nullable-типы (со знаком вопроса), data-классы и корутины. Старая рефлексия Java ничего об этом не знает.

Поэтому в Kotlin создали свою собственную обертку — **`KClass`** (и библиотеку `kotlin-reflect`).

* **Аналогия из жизни:** Медицинские снимки.
* `Class` (из Java) — это старый черно-белый рентгеновский снимок. Он показывает только "кости" — голые поля и методы, как их видит JVM. Он не покажет тебе, может ли поле быть `null`.
* `KClass` (из Kotlin) — это современный цветной снимок МРТ. Он содержит "метаданные" и показывает всю анатомию именно так, как ты написал её на языке Kotlin.



```kotlin
class User(val name: String, var age: Int?)

fun main() {
    val user = User("Алексей", 30)

    // 1. Получаем KClass (современный МРТ снимок Kotlin)
    // Используем оператор ::class
    val kClass = user::class // или User::class
    println("Имя Kotlin-класса: ${kClass.simpleName}")

    // 2. Получаем java.lang.Class (старый рентген Java)
    // Используем .java после ::class
    val javaClass = user::class.java // или user.javaClass
    println("Имя Java-класса: ${javaClass.name}")
}

```

### 2. Динамический анализ: Читаем свойства "вслепую"

Представь, что мы пишем библиотеку, которая должна превратить любой объект в строку формата JSON. Мы заранее не знаем, какой класс нам передадут: это может быть `User`, `Car` или `Document`.

С помощью библиотеки `kotlin-reflect` мы можем перебрать все свойства объекта и прочитать их значения динамически.

* **Аналогия из жизни:** Таможенный досмотр. Таможенник (наша функция) не знает, что лежит в закрытом контейнере (объекте). Но он берет официальный манифест (KClass), читает: *"Ага, тут есть свойство 'Вес'"*, и затем требует: *"А ну-ка, покажи мне значение этого 'Веса' для данного конкретного контейнера"*.

*(Важное примечание: для работы расширенных возможностей рефлексии в проект необходимо подключить зависимость `org.jetbrains.kotlin:kotlin-reflect`, так как она весит около 2.5 МБ и разработчики вынесли её из стандартной библиотеки, чтобы не утяжелять простые программы).*

```kotlin
// Подключаем необходимые классы из библиотеки kotlin-reflect
import kotlin.reflect.full.memberProperties
import kotlin.reflect.full.memberFunctions

data class Product(val id: Int, val title: String, val price: Double) {
    fun applyDiscount() {
        println("Скидка применена к $title")
    }
}

fun analyzeObject(obj: Any) {
    // Получаем МРТ-снимок (KClass) переданного неизвестного объекта
    val clazz = obj::class
    
    println("--- Анализ класса: ${clazz.simpleName} ---")

    // clazz.memberProperties возвращает список всех свойств класса.
    // Мы перебираем их в цикле.
    for (property in clazz.memberProperties) {
        
        // property.name - это имя переменной (например, "title")
        // property.returnType - это тип (например, "kotlin.String")
        val name = property.name
        val type = property.returnType
        
        // Магия рефлексии: мы ПРОСИМ свойство прочитать свое значение из конкретного объекта (obj)
        // Метод getter.call() динамически достает данные.
        val value = property.getter.call(obj)
        
        println("Свойство: $name, Тип: $type, Значение: $value")
    }
}

fun main() {
    val laptop = Product(101, "Ноутбук", 1500.0)
    
    // Передаем объект. Функция analyzeObject ничего не знала о классе Product на этапе написания кода!
    analyzeObject(laptop)
}

```

### 3. Динамический вызов методов

Точно так же мы можем не только читать данные, но и заставлять объект выполнять действия, зная только имя функции в виде обычного текста (String).

* **Аналогия из жизни:** Пульт оператора на заводе. На пульте есть клавиатура. Оператор печатает текст `"ЗАПУСТИТЬ_НАСОС"`. Система с помощью рефлексии ищет в чертежах завода (KClass) механизм с таким именем и нажимает на его рубильник.

```kotlin
fun callMethodByName(obj: Any, methodName: String) {
    val clazz = obj::class

    // Ищем среди всех функций класса ту, имя которой совпадает с переданной строкой
    val function = clazz.memberFunctions.find { it.name == methodName }

    if (function != null) {
        println("Функция '$methodName' найдена! Вызываю...")
        // Метод call() запускает найденную функцию. 
        // Первым аргументом всегда передается сам объект (receiver).
        function.call(obj) 
    } else {
        println("Ошибка: Функция '$methodName' не найдена в классе ${clazz.simpleName}")
    }
}

fun main() {
    val laptop = Product(101, "Ноутбук", 1500.0)
    
    // Мы вызываем метод, передав его имя как обычный текст!
    callMethodByName(laptop, "applyDiscount") 
    
    // Попытка вызвать несуществующий метод
    callMethodByName(laptop, "flyToSpace")
}

```

**Осторожно: Цена рефлексии.**
Рефлексия работает очень медленно по сравнению с обычным кодом. Вызов `function.call()` может быть в десятки раз медленнее, чем прямой вызов `laptop.applyDiscount()`. Поэтому рефлексию используют только при написании гибких фреймворков и библиотек, где без неё невозможно обойтись, но её избегают в обычной бизнес-логике, требующей высокой производительности.

---

Итог:

* **Рефлексия** — это механизм, позволяющий программе анализировать свою собственную структуру (классы, свойства, методы) и взаимодействовать с ней динамически во время выполнения (в рантайме).
* Из-за того, что Kotlin имеет уникальные особенности (свойства, Nullability), которых нет в JVM, он предоставляет собственный класс метаданных **`KClass`** (получаемый через `::class`).
* Старый Java-класс метаданных `java.lang.Class` по-прежнему доступен (через `::class.java`) и необходим для совместимости с Java-библиотеками.
* Для глубокого анализа Kotlin-кода требуется отдельная библиотека `kotlin-reflect`. Она предоставляет мощные инструменты: например, коллекции `memberProperties` (для чтения свойств объекта вслепую) и `memberFunctions` (для поиска и динамического вызова методов по их строковому имени).
* Рефлексия является фундаментом для библиотек сериализации (JSON) и контейнеров инъекции зависимостей (DI), но требует дополнительных затрат оперативной памяти и процессорного времени, поэтому её не следует применять для решения тривиальных задач.
