## Глава 35. Каналы (Channels)

### 1. Модель CSP: Общайся, чтобы разделять память

В классическом многопоточном программировании (Java, C++) правит бал концепция *Shared Mutable State* (Общее изменяемое состояние), которую мы разбирали в модуле 6.4. Чтобы потоки не передрались за одну переменную, мы ставим `Mutex` или `synchronized`.

В Kotlin (и в языке Go, откуда эта идея была популяризована) предлагается совершенно другой архитектурный подход: **CSP (Communicating Sequential Processes — Взаимодействующие последовательные процессы)**.

Девиз CSP звучит так:

> *"Не общайтесь, разделяя память. Вместо этого разделяйте память, общаясь"*.

* **Аналогия из жизни:** Пневмопочта в старом банке.
Вместо того чтобы десять кассиров бегали в одно хранилище и толкались там у сейфа (Shared Memory), каждый кассир сидит в своей изолированной комнате. Если кассиру нужны деньги, он пишет записку, кладет ее в капсулу и отправляет по трубе пневмопочты (Channel) главному банкиру. Банкир обрабатывает записки строго по очереди. Никаких гонок данных.

### 2. Механика Канала: Труба с двумя концами

Канал в Kotlin — это интерфейс `Channel`, который реализует блокирующую (suspend) очередь. У него есть два главных метода:

* `send(value)` — положить элемент в трубу.
* `receive()` — достать элемент из трубы.

По умолчанию канал является **Rendezvous** (Точка рандеву). У него **нет буфера** (вместимость = 0). Это значит, что `send` немедленно уснет (`suspend`), пока на другом конце кто-нибудь не вызовет `receive`, чтобы забрать значение прямо из рук в руки. И наоборот: `receive` уснет, если труба пуста.

```kotlin
import kotlinx.coroutines.*
import kotlinx.coroutines.channels.Channel

fun main() = runBlocking {
    val channel = Channel<String>() // Создаем трубу

    // Корутина 1: Отправитель (Producer)
    launch {
        println("Отправитель: Пытаюсь отправить данные...")
        channel.send("Секретное сообщение") // Заснет здесь, пока кто-то не заберет!
        println("Отправитель: Фух, забрали. Иду дальше.")
    }

    delay(1000) // Ждем секунду

    // Корутина 2: Получатель (Consumer)
    launch {
        println("Получатель: Заглядываю в трубу...")
        val msg = channel.receive() // Достаем из трубы
        println("Получатель: Прочитал '$msg'")
    }
}

```

*(Справка: Если при создании канала указать `Channel(10)`, появится буфер на 10 элементов, и отправитель сможет закинуть 10 сообщений без сна, даже если их никто не забирает).*

### 3. Паттерн: Воркер-пул (Worker Pool / Fan-out)

Каналы идеально подходят для распределения тяжелой работы. Если у нас есть миллион задач, мы не хотим запускать миллион корутин (это съест процессор). Мы хотим создать пул из 5 "рабочих" (воркеров), которые будут брать задачи из одной трубы по мере освобождения.

Этот паттерн называется **Fan-out** (Разветвление). Несколько корутин-получателей одновременно слушают один и тот же канал. Сообщение достанется тому, кто первым успеет сделать `receive()`.

* **Аналогия из жизни:** Лента выдачи багажа или конвейер. По конвейеру (Каналу) едут посылки. Вокруг конвейера стоят 3 грузчика (Воркера). Как только посылка подъезжает, свободный грузчик хватает её и несет на склад. Остальные ждут следующую. Одна посылка никогда не достанется двум грузчикам одновременно.

```kotlin
// Функция-воркер. Она просто бесконечно слушает свой канал (пока он не закроется)
fun CoroutineScope.launchWorker(id: Int, tasksChannel: ReceiveChannel<String>) = launch {
    for (task in tasksChannel) { // Цикл for автоматически достает данные (receive)
        println("👷 Воркер #$id взял в работу: $task")
        delay(500) // Имитация тяжелой работы
        println("✅ Воркер #$id закончил: $task")
    }
}

fun main() = runBlocking {
    val tasksChannel = Channel<String>(capacity = 10) // Труба с буфером

    // Запускаем пул из 3 воркеров. Они мгновенно "уснут", ожидая задачи.
    repeat(3) { workerId ->
        launchWorker(workerId, tasksChannel)
    }

    // Главный процесс генерирует 10 задач и кидает в трубу
    launch {
        for (i in 1..10) {
            tasksChannel.send("Задача_$i")
        }
        // Закрываем канал. Это сигнал воркерам: "Задач больше нет, можете расходиться".
        tasksChannel.close() 
    }
}

```

*Результат:* Задачи будут равномерно распределены между тремя воркерами. Как только воркер освобождается, он берет следующую задачу из канала. Это идеальный балансировщик нагрузки (Load Balancer)!

### 4. Паттерн Inbox / Outbox (Конвейерная обработка)

Когда система становится сложной (например, парсинг данных, конвертация, сжатие и отправка на сервер), логику разбивают на независимые этапы.

Паттерн **Inbox/Outbox** (Входящие/Исходящие) — это когда функция ничего не знает о внешнем мире. Она принимает один канал (`inbox` — откуда брать сырые данные) и второй канал (`outbox` — куда выплевывать готовый результат).

* **Аналогия из жизни:** Заводской цех контроля качества. У работника есть корзина "Входящие детали" (Inbox) и корзина "Проверенные детали" (Outbox). Он берет деталь слева, проверяет, кладет направо. Ему абсолютно всё равно, кто принес детали слева и кто заберет детали справа.

```kotlin
data class RawImage(val name: String)
data class CompressedImage(val name: String, val size: Int)

// Обработчик знает ТОЛЬКО о двух трубах: Inbox и Outbox
fun CoroutineScope.launchImageCompressor(
    inbox: ReceiveChannel<RawImage>, 
    outbox: SendChannel<CompressedImage>
) = launch {
    for (raw in inbox) { // Достаем из Входящих
        println("📦 Сжимаем: ${raw.name}...")
        delay(300) // Имитация сжатия
        
        val result = CompressedImage(raw.name, size = 50)
        outbox.send(result) // Кидаем в Исходящие
    }
}

fun main() = runBlocking {
    // Создаем две трубы (конвейерные ленты)
    val rawImagesChannel = Channel<RawImage>()
    val compressedImagesChannel = Channel<CompressedImage>()

    // 1. Запускаем компрессор. Он соединяет две трубы.
    launchImageCompressor(inbox = rawImagesChannel, outbox = compressedImagesChannel)

    // 2. Корутина-фотограф кидает фотки в первую трубу (Inbox)
    launch {
        rawImagesChannel.send(RawImage("Фото_отпуска.png"))
        rawImagesChannel.send(RawImage("Скриншот.png"))
        rawImagesChannel.close() // Фотографий больше нет
    }

    // 3. Корутина-сервер слушает вторую трубу (Outbox) и выводит результат
    launch {
        for (compressed in compressedImagesChannel) {
            println("🌐 Загружено на сервер: $compressed")
        }
    }
}

```

**Зачем это нужно?**
Такая архитектура позволяет строить **Pipelines (Конвейеры)** бесконечной сложности. Вы можете в любой момент добавить между этими этапами еще одну корутину (например, наложение водяных знаков), и вам не придется переписывать старый код! Вы просто соедините их новыми "трубами".

---

Итог:

* **Модель CSP** — архитектурный подход, при котором потоки не блокируют друг друга в борьбе за общую память, а обмениваются сообщениями по выделенным линиям связи.
* **Channels (Каналы)** в Kotlin — это потокобезопасные асинхронные очереди для передачи данных между корутинами. Основные методы: `send()` (положить) и `receive()` (достать). Оба метода являются `suspend` функциями и могут приостанавливать выполнение, если канал переполнен или пуст.
* **Паттерн Воркер-пул (Worker Pool / Fan-out)** — создание нескольких корутин, которые одновременно читают данные из одного канала (`ReceiveChannel`). Это нативный механизм балансировки нагрузки: сообщение гарантированно забирает только один свободный воркер.
* **Паттерн Inbox/Outbox** — изоляция бизнес-логики корутины. Функция принимает интерфейс `ReceiveChannel` для получения данных и `SendChannel` для выдачи результата. Это позволяет строить гибкие, слабосвязанные архитектурные конвейеры (Pipelines) любой глубины.
