## Глава 25. Value / Inline классы

### 1. Проблема: Одержимость примитивами (Primitive Obsession)

В программировании есть антипаттерн "одержимость примитивами". Это когда мы используем базовые типы (строки, числа) для сущностей, которые несут в себе важный бизнес-смысл.

Например, пароль — это `String`. И email — это `String`. Время в миллисекундах — это `Int`, и время в секундах — тоже `Int`.

* **Аналогия из жизни:** Соль и Сахар. И то, и другое — это белый порошок в прозрачной банке (тип `String`). Если на банках нет строгих этикеток, ты можешь случайно насыпать соль в чай. Физически порошок идеально растворится (код скомпилируется), но результат будет катастрофическим (бизнес-ошибка).

```kotlin
fun authUser(email: String, password: String) {
    println("Авторизуем $email...")
}

fun main() {
    val myEmail = "test@mail.com"
    val myPass = "qwerty123"

    // ОШИБКА! Мы случайно перепутали аргументы местами.
    // Но для компилятора это две обычные строки. 
    // Код успешно скомпилируется, а в продакшене пользователь не сможет войти.
    authUser(myPass, myEmail) 
}

```

Чтобы защититься от этого, программисты на Java создают классы-обертки: `class Email(val value: String)`. Но мы уже знаем из прошлого модуля, что создание объектов в куче (Heap) — это трата оперативной памяти и нагрузка на сборщик мусора. Если мы будем оборачивать каждый пароль и каждую миллисекунду в тяжелый класс, наша программа начнет тормозить.

### 2. Решение: Value-классы (Синтаксис и Магия)

Kotlin предлагает идеальное решение: **`value class`** (ранее они назывались `inline class`).

Это класс, который дает строгую защиту типов на этапе написания кода, но **полностью исчезает в рантайме** (во время работы программы). Это называется *Zero-cost abstraction* (абстракция с нулевой стоимостью).

* **Аналогия из жизни:** Голографическая этикетка на банке с порошком. На этапе подготовки (написания кода) повар видит яркую голограмму «Это СОЛЬ!» и физически не может перепутать банки. Но как только он высыпает содержимое в суп (программа запускается), голограмма исчезает. В суп попадает только чистый порошок, без куска пластика или картона (без выделения памяти под объект-обертку).

Чтобы создать такой класс, нужно написать `value class` и обязательно добавить аннотацию `@JvmInline` (требование совместимости с виртуальной машиной Java).

**Главное правило:** У `value`-класса в первичном конструкторе может быть **строго одно свойство** (тот самый примитив, который мы оборачиваем).

```kotlin
// Создаем строгие типы, оборачивая обычные строки
@JvmInline
value class Email(val value: String)

@JvmInline
value class Password(val value: String)

// Теперь функция требует не просто "строки", а конкретные бизнес-типы
fun loginUser(email: Email, password: Password) {
    // Внутри мы можем легко достать оригинальное значение, если нужно:
    println("Подключение к базе по почте: ${email.value}")
}

fun main() {
    val email = Email("admin@test.com")
    val pass = Password("secret")

    // loginUser(pass, email) // ОШИБКА КОМПИЛЯЦИИ! 
    // Компилятор бьет по рукам: "Нельзя передать Password туда, где ожидается Email".
    
    // Правильный вызов
    loginUser(email, pass)
}

```

### 3. Как это выглядит под капотом JVM?

Вся красота `value`-классов раскрывается после компиляции программы в байткод.

Компилятор Kotlin проводит строгие проверки, а затем "распаковывает" наши объекты. В байткоде (в рантайме JVM) функция `loginUser` превращается в обычную `loginUser(String, String)`, а переменная `email` становится просто `String`.

В оперативной памяти не создается никаких объектов `Email` или `Password`. Память расходуется ровно так же, как если бы мы использовали голые примитивы, но при этом наш код на 100% защищен от человеческого фактора!

### 4. Отличие от `typealias` (Важный архитектурный нюанс)

В модуле 3.4 мы изучали `typealias` (синонимы типов). Может показаться, что они делают то же самое, но разница между ними колоссальна.

* **`typealias` — это просто псевдоним (кличка).**
Если ты напишешь `typealias Email = String`, компилятор *не создаст новый тип*. Для него `Email` и `String` — это одно и то же. Он **не защитит** тебя от того, чтобы передать пароль вместо почты.
* **`value class` — это новый, уникальный тип данных.**
Компилятор строго следит за тем, чтобы ты не смешивал его ни со строками, ни с другими `value`-классами. Присвоить обычную `String` в переменную `Email` без явного вызова конструктора `Email("...")` невозможно.

### 5. Полноценный класс без памяти

Несмотря на то, что `value`-класс исчезает в памяти, до момента компиляции он ведет себя как полноценный класс. Мы можем писать внутри него методы и вычисляемые свойства.

* **Аналогия из жизни:** Умные очки (AR). Для внешнего мира (для оперативной памяти) ты просто человек. Очков не существует физически. Но в твоем поле зрения (внутри кода) у тебя есть интерфейс, методы расчета траектории и подсказки.

```kotlin
@JvmInline
value class TemperatureCelsius(val value: Double) {
    
    // Мы можем добавить методы прямо в value class!
    // В памяти этого метода не будет, компилятор просто "встроит" математику.
    fun toFahrenheit(): Double {
        return (value * 9 / 5) + 32
    }
    
    // И даже свойства (но только вычисляемые, без скрытого поля field)
    val isFreezing: Boolean
        get() = value <= 0.0
}

fun main() {
    val temp = TemperatureCelsius(-5.0)
    
    if (temp.isFreezing) {
        println("Холодно! В фаренгейтах это: ${temp.toFahrenheit()}")
    }
}

```

---

Итог:

* **Одержимость примитивами** — это архитектурная проблема, когда для хранения бизнес-данных (пароли, ID, различные единицы измерения) используются одинаковые примитивные типы (`String`, `Int`), что ведет к багам из-за случайной подмены аргументов.
* **Value-классы** (ранее `inline class`) объявляются с помощью аннотации `@JvmInline value class` и служат для создания строгих типов, защищающих от путаницы.
* Главное требование: у `value`-класса должно быть **строго одно свойство** в первичном конструкторе (`val`).
* На этапе компиляции `value`-класс обеспечивает жесткую типизацию (нельзя передать `Password` вместо `Email`). Но в рантайме компилятор стирает этот класс, оставляя только внутренний примитив. Это позволяет избежать аллокации памяти (Heap Allocation) под объекты-обертки.
* В отличие от `typealias` (который просто дает старым типам новые имена, не защищая от ошибок), `value class` создает полноценный новый тип, несовместимый с оригинальным примитивом без явной конвертации.
* Внутри `value`-классов можно объявлять методы и вычисляемые свойства (геттеры), которые также компилируются без выделения лишней памяти.
