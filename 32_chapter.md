## Глава 32. Примитивы синхронизации

### 1. Почему `synchronized` — это зло для корутин?

Блок `synchronized` (или `ReentrantLock` из Java) — это механизм, который работает на уровне потоков операционной системы. Если один поток зашел в блок `synchronized`, а второй пытается туда попасть, второй поток **физически блокируется (усыпляется операционной системой)**, пока первый не выйдет.

В корутинах мы используем крошечный пул потоков (например, в `Dispatchers.Default` потоков столько же, сколько ядер процессора). Если у тебя 4 ядра, и 4 корутины упрутся в блок `synchronized`, они заблокируют все 4 потока! Твое приложение полностью зависнет, хотя 100 000 других корутин были готовы выполнять другую работу.

* **Аналогия из жизни:** Кабинка туалета в офисе. `synchronized` — это когда человек дергает закрытую ручку и **остается стоять прямо под дверью**, блокируя весь коридор и не давая никому пройти, пока дверь не откроется.

### 2. Решение 1: `Mutex` (Взаимоисключающая блокировка)

Вместо блокировки потоков Kotlin предлагает **Mutex (Mutual Exclusion)**. Это корутино-совместимый "замок".

Главное отличие `Mutex` в том, что его метод `.lock()` является `suspend` функцией. Если замок занят, корутина не блокирует поток процессора. Она просто "засыпает" (освобождает поток для других задач) и просит: *"Разбудите меня, когда замок освободится"*.

* **Аналогия из жизни:** Примерочная в магазине одежды с электронной очередью. Если примерочная занята, ты берешь номерок и идешь гулять по торговому центру (освобождаешь коридор). Когда примерочная освобождается, тебе на телефон приходит уведомление, и ты возвращаешься.

```kotlin
import kotlinx.coroutines.*
import kotlinx.coroutines.sync.Mutex
import kotlinx.coroutines.sync.withLock

class BankAccount {
    var balance = 0.0
        private set
        
    // Создаем Mutex для защиты баланса
    private val mutex = Mutex()

    suspend fun deposit(amount: Double) {
        // withLock автоматически делает .lock() на входе и .unlock() на выходе
        // Даже если внутри вылетит ошибка (Exception), замок всё равно откроется!
        mutex.withLock {
            // Только одна корутина может находиться в этом блоке одновременно
            val current = balance
            delay(10) // Имитация сложной сетевой логики
            balance = current + amount
        }
    }
}

fun main() = runBlocking {
    val account = BankAccount()
    
    // Запускаем 1000 корутин, которые одновременно пытаются положить по 1 рублю
    val jobs = List(1000) {
        launch(Dispatchers.Default) {
            account.deposit(1.0)
        }
    }
    
    jobs.joinAll() // Ждем завершения всех корутин
    println("Итоговый баланс: ${account.balance}") // Гарантированно выведет 1000.0
}

```

### 3. Решение 2: Семафоры (`Semaphore`)

Семафор работает точно так же, как `Mutex` (он тоже "усыпляет" корутины, а не блокирует потоки), но с одним отличием: он позволяет зайти внутрь не одной корутине, а **строго заданному числу $N$**.
*(Кстати, `Mutex` — это просто `Semaphore(1)` под капотом).*

* **Аналогия из жизни:** Эксклюзивный ресторан. В нем всего 3 столика. Хостес (Семафор) пускает внутрь первых трех гостей. Четвертого она вежливо просит подождать на улице (suspend). Как только любой из трех столиков освобождается, хостес пускает четвертого гостя внутрь.

Семафоры идеально подходят для ограничения нагрузки (Rate Limiting). Например, если ты не хочешь случайно "положить" чужой сервер, отправляя ему 10 000 запросов одновременно.

```kotlin
import kotlinx.coroutines.sync.Semaphore
import kotlinx.coroutines.sync.withPermit

class NetworkService {
    // Разрешаем не более 5 одновременных сетевых запросов
    private val semaphore = Semaphore(5)

    suspend fun fetchData(id: Int): String {
        // Если 5 корутин уже здесь, 6-я "уснет" и будет ждать
        return semaphore.withPermit {
            println("Загрузка данных для $id...")
            delay(1000) // Имитация долгого запроса
            "Результат $id"
        }
    }
}

```

### 4. Решение 3: Паттерн Actor (Актор)

`Mutex` отлично работает, но у него есть минус: программист может банально забыть обернуть критический кусок кода в `mutex.withLock { ... }`, и тогда вся защита рухнет.

Более продвинутый, архитектурный подход называется **Actor Model**.
Идея гениальна в своей простоте: мы вообще **запрещаем** всем корутинам напрямую трогать изменяемое состояние (переменную). Мы прячем это состояние внутри одной-единственной "особой" корутины (Актора). Остальные корутины могут общаться с ней только путем отправки сообщений в специальную трубу — `Channel` (Канал).

* **Аналогия из жизни:** Касса в банке. Деньги банка — это общее изменяемое состояние. Клиентам банка (обычным корутинам) строго запрещено самим лезть в кассу, иначе начнется драка (гонка данных). Вместо этого банк нанимает Кассира (Актора). Кассир сидит за бронированным стеклом. Клиенты пишут свои запросы на бумажках (Сообщения) и по одному просовывают в щель под стеклом (Канал). Кассир берет бумажки строго по очереди и меняет баланс. Драка физически невозможна.

```kotlin
import kotlinx.coroutines.channels.Channel

// 1. Описываем типы сообщений (записок), которые понимает наш Актор
sealed class CounterMsg {
    object Increment : CounterMsg()
    class GetCount(val responseChannel: CompletableDeferred<Int>) : CounterMsg()
}

// 2. Создаем класс-Актор
class CounterActor(scope: CoroutineScope) {
    
    // Канал - это "щель под стеклом", куда прилетают записки
    val mailbox = Channel<CounterMsg>()

    init {
        // Запускаем Кассира (корутину), которая будет вечно жить и читать почту
        scope.launch {
            var counter = 0 // Наше состояние. Оно спрятано здесь! Никаких Mutex не нужно.
            
            // Читаем записки СТРОГО по одной (последовательно)
            for (msg in mailbox) {
                when (msg) {
                    is CounterMsg.Increment -> counter++
                    is CounterMsg.GetCount -> msg.responseChannel.complete(counter)
                }
            }
        }
    }
}

```

**Когда что использовать?**

* Если у тебя простой счетчик или список, который обновляется изредка — бери `Mutex`. Это просто и быстро.
* Если тебе нужно ограничить пропускную способность (например, 10 подключений к базе) — бери `Semaphore`.
* Если у тебя сложный объект со множеством состояний и запутанной логикой обновления (Enterprise уровень) — изолируй его через паттерн `Actor`, это избавит тебя от головной боли с "забытыми" блокировками.

---

Итог:

* Использование классических Java-блокировок (`synchronized`, `ReentrantLock`) в Kotlin корутинах недопустимо, так как они физически усыпляют потоки операционной системы, что приводит к зависанию (голоданию) пула потоков диспетчера.
* **`Mutex`** — это корутино-безопасная альтернатива блокировкам. Вместо блокировки потока метод `lock()` приостанавливает (suspend) текущую корутину до освобождения ресурса. Для безопасности всегда используется обертка `withLock { }`.
* **`Semaphore`** — аналог Mutex, но позволяющий одновременно выполнять критическую секцию заданному количеству корутин ($N$). Используется для ограничения параллельной нагрузки (Rate Limiting).
* **Паттерн Actor** — архитектурный подход для защиты данных. Он подразумевает инкапсуляцию изменяемого состояния внутри одной отдельной корутины. Внешние корутины не имеют прямого доступа к памяти, а отправляют "сообщения" (намерения) в `Channel`. Поскольку Актор обрабатывает сообщения из канала последовательно, гонка данных становится невозможной архитектурно.
