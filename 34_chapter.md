## Глава 34. StateFlow и SharedFlow

### 1. `SharedFlow`: Радиовещание (Трансляция событий)

`SharedFlow` (Разделяемый поток) — это классический горячий поток. Он существует и работает (эмитит данные) независимо от наличия подписчиков. Главная его фишка в том, что он **разделяет (share) одну и ту же трансляцию** между всеми подписчиками.

* **Аналогия из жизни:** Прямой эфир по радио. Радиостанция вещает музыку в эфир, даже если никто не включил приемник (данные уходят в никуда). Если ты включишь радио в 14:05, ты начнешь слушать эфир именно с этого момента. Ты пропустил всё, что играло до 14:05. Если твой друг включит радио в это же время на своей частоте, вы будете слушать **одну и ту же** песню синхронно.

`SharedFlow` идеально подходит для **одноразовых событий** (One-time events): показать всплывающее уведомление (Snackbar), проиграть звук, перекинуть на другой экран.

```kotlin
import kotlinx.coroutines.*
import kotlinx.coroutines.flow.*

fun main() = runBlocking {
    // Создаем горячий поток
    val sharedFlow = MutableSharedFlow<String>()

    // Запускаем корутину, которая генерирует события (вещает в эфир)
    launch {
        println("🎙️ Радиостанция начала вещание...")
        sharedFlow.emit("Песня 1") // Уйдет в никуда, подписчиков еще нет
        delay(100)
        sharedFlow.emit("Песня 2")
    }

    delay(50) // Ждем немного, чтобы "Песня 1" уже прошла

    // Подписчик 1 (Включил радио)
    launch {
        sharedFlow.collect { println("Слушатель 1 услышал: $it") }
    }

    // Подписчик 2 (Включил радио)
    launch {
        sharedFlow.collect { println("Слушатель 2 услышал: $it") }
    }
}
// Вывод: 
// Слушатель 1 услышал: Песня 2
// Слушатель 2 услышал: Песня 2

```

**Магия `replay` (Кэш эфира):**
При создании `MutableSharedFlow(replay = 2)` ты можешь указать параметр `replay`. Это кэш. Если слушатель подключится поздно, радиостанция мгновенно "прокрутит" ему 2 последние песни из кэша, прежде чем он начнет слушать прямой эфир.

### 2. `StateFlow`: Информационное табло (Хранение состояния)

`StateFlow` — это специализированная, урезанная версия `SharedFlow`, созданная строго для одной цели: **хранить и показывать текущее состояние (State)**.

У него есть три жестких отличия:

1. У него **всегда** есть начальное значение (`MutableStateFlow(0)`).
2. У него всегда есть свойство `.value`, к которому можно обратиться напрямую, без подписки (как к обычной переменной).
3. **Конфляция (Слияние одинаковых значений):** Если ты попытаешься отправить туда значение "А", а потом снова "А", `StateFlow` проигнорирует второе отправление. Подписчики не получат уведомление, если состояние фактически не изменилось.

* **Аналогия из жизни:** Электронное табло со счетом на футбольном стадионе.
* У него всегда есть начальное значение (0:0).
* Ты можешь в любую секунду просто посмотреть на табло и узнать текущий счет (`state.value`), тебе не нужно "подписываться" на него.
* Если счет 1:0, и оператор случайно еще раз нажмет кнопку "Сделать 1:0", табло не будет моргать и кричать "Внимание, счет изменился на 1:0!". Состояние осталось прежним.



`StateFlow` идеально подходит для UI: хранить текущий текст в поле ввода, состояние переключателя, список загруженных данных.

```kotlin
fun main() = runBlocking {
    // Обязательно передаем начальное состояние
    val stateFlow = MutableStateFlow("Загрузка...") 

    launch {
        stateFlow.collect { println("UI показывает: $it") }
    }

    delay(50)
    stateFlow.value = "Данные получены!" // Обновляем состояние
    delay(50)
    
    // Пытаемся записать то же самое значение.
    // StateFlow проигнорирует это! UI не будет перерисовываться впустую.
    stateFlow.value = "Данные получены!" 
}

```

### 3. Backpressure (Давление потока)

В реактивном программировании есть фундаментальная проблема: **Что делать, если источник генерирует данные быстрее, чем подписчик успевает их обрабатывать?** Эта проблема называется *Backpressure* (Обратное давление).

* **Аналогия из жизни:** Шеф-повар (Источник) и Официант (Подписчик). Шеф готовит бургеры со скоростью 1 бургер в секунду. Официант хромает и относит бургер клиенту за 5 секунд. Возникает "давление". Кухня заваливается бургерами.

Как Kotlin решает эту проблему в горячих потоках? С помощью параметра `onBufferOverflow` (стратегия переполнения буфера) при создании `SharedFlow`. У нас есть 3 варианта:

1. **`SUSPEND` (Приостановка) — По умолчанию:**
Если буфер переполнен, функция `emit()` у шеф-повара заблокируется (`suspend`). Шеф буквально "уснет" с бургером в руках и не будет готовить следующий, пока официант не вернется и не заберет тарелку.
2. **`DROP_OLDEST` (Выбросить старое):**
Шеф продолжает готовить на максимальной скорости. Когда на столе не остается места, он берет самый первый (остывший) бургер, выбрасывает его в мусорку и кладет на его место свежий. Официант всегда забирает самые свежие данные, но мы теряем часть истории. *(Кстати, `StateFlow` под капотом использует именно эту стратегию с буфером размером 1).*
3. **`DROP_LATEST` (Выбросить новое):**
Шеф продолжает готовить, но если стол занят, он просто выбрасывает свежеприготовленный бургер в мусорку. Он не будет заменять им старые бургеры, которые ждут официанта.

---

Итог:

* **`SharedFlow`** — это горячий поток для широковещательной трансляции событий (multicasting). Он не имеет начального состояния и активен вне зависимости от наличия коллекторов. Отлично подходит для событий (навигация, ошибки, снекбары).
* Параметр `replay` в `SharedFlow` задает размер кэша, который будет мгновенно передан (воспроизведен) любому новому подписчику в момент подключения.
* **`StateFlow`** — это специализированный горячий поток для хранения единичного состояния. У него всегда есть `value`, он требует инициализации начальным значением и автоматически игнорирует (conflates) дубликаты идущих подряд значений. Отлично подходит для хранения стейта UI (ViewModel State).
* **Backpressure (Обратное давление)** возникает, когда эмиттер работает быстрее коллектора. В `SharedFlow` эта проблема решается настройкой буфера (`extraBufferCapacity`) и стратегией `onBufferOverflow` (`SUSPEND` эмиттера, `DROP_OLDEST` или `DROP_LATEST` значений).
