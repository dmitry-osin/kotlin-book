## Глава 6. Data-классы и Деструктуризация

### 1. Проблема обычных классов и магия `data class`

Давай представим, что мы создали обычный класс для хранения информации о билете на поезд.

* **Аналогия из жизни:** Представь, что ты распечатал два абсолютно одинаковых билета на один и тот же поезд, в один и тот же вагон, на одно и то же место. Физически — это два разных куска бумаги. Если ты спросишь обычный язык программирования: "Это один и тот же билет?", он сравнит их *адреса в памяти* (физические листы бумаги) и ответит: "Нет, они разные".

Именно так ведут себя обычные классы. Но для бизнес-логики нам важно *содержимое*, а не адреса в памяти!

Добавив приставку `data`, мы приказываем компилятору автоматически сгенерировать правильные методы `equals()` (для сравнения содержимого), `hashCode()` (для быстрой работы в коллекциях) и `toString()` (для красивого вывода в консоль).

```kotlin
// Обычный класс
class StandardTicket(val trainNumber: String, val seat: Int)

// Дата-класс (Data class)
data class DataTicket(val trainNumber: String, val seat: Int)

fun main() {
    val ticket1 = StandardTicket("Сапсан-2000", 42)
    val ticket2 = StandardTicket("Сапсан-2000", 42)

    // Сравниваем обычные классы (оператор == в Kotlin вызывает метод equals)
    // Результат: false. Компилятор видит два разных "куска бумаги" в памяти.
    println(ticket1 == ticket2) 

    val dataTicket1 = DataTicket("Сапсан-2000", 42)
    val dataTicket2 = DataTicket("Сапсан-2000", 42)

    // Сравниваем дата-классы
    // Результат: true. Компилятор посимвольно сравнил их свойства. Содержимое идентично!
    println(dataTicket1 == dataTicket2) 
    
    // toString() тоже сгенерирован автоматически и читается человеком:
    // Выведет: DataTicket(trainNumber=Сапсан-2000, seat=42)
    println(dataTicket1)
}

```

### 2. Метод `copy()`: Безопасное изменение неизменяемого

В первом модуле мы усвоили правило: старайся всегда использовать `val` (неизменяемые переменные). Но что делать, если данные всё-таки нужно немного изменить?

* **Аналогия из жизни:** У тебя есть подписанный договор аренды (неизменяемый объект `val`). Тебе нужно изменить в нем только дату заезда, оставив все остальные 50 пунктов без изменений. Ты же не будешь переписывать весь договор с нуля? Ты делаешь ксерокопию старого договора, меняешь одну строчку и ставишь новую подпись.

Именно для этого компилятор генерирует в `data`-классах метод `copy()`. Он создает точный клон объекта, позволяя "на лету" заменить нужные свойства.

```kotlin
data class HotelBooking(val city: String, val hotelName: String, val days: Int, val isPaid: Boolean)

fun main() {
    // Создаем бронь. Она полностью неизменяемая (все свойства val).
    val myBooking = HotelBooking("Москва", "Космос", 3, false)

    // Клиент решил оплатить бронь и продлить ее на 5 дней.
    // Мы не можем написать myBooking.days = 5 (ошибка компиляции).
    // Вместо этого мы создаем НОВУЮ бронь на основе старой через copy()
    val updatedBooking = myBooking.copy(days = 5, isPaid = true)

    // Исходный объект остался нетронутым (безопасно для других частей программы),
    // а у нас на руках новый актуальный объект.
}

```

### 3. Деструктуризация и функции `componentN`

Часто бывает нужно достать данные из объекта и разложить их по отдельным переменным. В Kotlin это делается одной элегантной строчкой с помощью **деструктуризации**.

* **Аналогия из жизни:** Покупка "Комбо-обеда" в фастфуде. На кассе тебе выдают единый пакет (объект дата-класса). Ты садишься за стол и распаковываешь его: отдельно ставишь бургер, отдельно — картошку, отдельно — напиток. Теперь ты взаимодействуешь с ними как с независимыми предметами.

Как это работает под капотом? Когда ты создаешь `data class`, компилятор невидимо генерирует для каждого свойства из первичного конструктора специальные функции: `component1()`, `component2()`, `component3()` и так далее — строго по порядку их объявления.

```kotlin
// Дата-класс, описывающий комбо-обед
data class ComboMeal(val mainDish: String, val sideDish: String, val drink: String)

fun main() {
    val myLunch = ComboMeal("Чизбургер", "Картофель фри", "Кола")

    // --- Классический подход (без деструктуризации) ---
    val m = myLunch.mainDish
    val s = myLunch.sideDish
    val d = myLunch.drink

    // --- Подход Kotlin (Деструктуризация) ---
    // Мы объявляем несколько переменных в скобках. 
    // Kotlin автоматически вызывает component1(), component2() и component3() 
    // и раскладывает значения по переменным строго по порядку!
    val (burger, fries, soda) = myLunch

    println("Я буду есть $burger и запивать $soda")
    
    // Если какая-то часть комбо нам не нужна, мы можем проигнорировать её 
    // с помощью символа нижнего подчеркивания (_), чтобы не тратить память:
    val (onlyBurger, _, onlyDrink) = myLunch
}

```

*Важное замечание: функции `componentN` привязаны именно к **порядку** свойств в конструкторе, а не к их именам. Если ты поменяешь местами `mainDish` и `drink` в конструкторе дата-класса, деструктуризация в старом коде отработает неверно (в переменную `burger` попадет напиток).*

---

Итог:

* `data class` создан для автоматизации написания классов-контейнеров данных. Добавление слова `data` избавляет от написания шаблонного кода.
* Компилятор автоматически генерирует для `data`-классов методы `equals()` и `hashCode()` (которые сравнивают объекты по содержимому полей, а не по ссылкам в памяти), а также читаемый метод `toString()`.
* Автоматически генерируемый метод `copy()` позволяет безопасно и быстро создавать клоны неизменяемых объектов (`val`), точечно перезаписывая только необходимые свойства.
* Деструктуризация (`val (a, b) = object`) позволяет распаковать свойства объекта в отдельные переменные в одну строку.
* Под капотом деструктуризация работает благодаря автогенерируемым методам `component1()`, `component2()` и т.д., которые возвращают свойства строго в том порядке, в котором они объявлены в первичном конструкторе.
