## Глава 5. Классы и Свойства

### 1. Классы и Конструкторы (Primary и Secondary)

В Kotlin создание класса и его инициализация максимально сжаты. У нас есть два типа конструкторов: первичный (Primary) и вторичные (Secondary).

#### Первичный конструктор (Primary Constructor)

Он объявляется прямо в "шапке" класса. Если ты используешь слова `val` или `var` прямо в скобках, Kotlin автоматически создает свойства класса и присваивает им переданные значения.

* **Аналогия из жизни:** Типовой проект дома. У архитектора есть жесткий чертеж (первичный конструктор), где четко сказано: нужно передать количество окон и цвет крыши. Без этих данных дом не построить.

```kotlin
// Первичный конструктор находится прямо в объявлении класса.
// val name и var age мгновенно становятся свойствами этого класса.
class User(val name: String, var age: Int) {
    
    // Блок init выполняется сразу при создании объекта (как часть первичного конструктора).
    // Здесь можно написать логику валидации.
    init {
        require(age >= 0) { "Возраст не может быть отрицательным" }
        println("Пользователь $name успешно создан.")
    }
}

```

#### Вторичный конструктор (Secondary Constructor)

Иногда нам нужны альтернативные способы создания объекта. Для этого внутри класса пишутся вторичные конструкторы (через слово `constructor`). **Главное правило:** любой вторичный конструктор обязан в конечном итоге вызвать первичный (делегировать ему работу).

* **Аналогия из жизни:** Клиент приходит к застройщику и говорит: "Я не знаю, сколько окон мне нужно, у меня просто есть 5 миллионов рублей" (вторичный конструктор). Менеджер берет эти деньги, сам высчитывает по прайсу количество окон и передает эти точные данные главному архитектору в типовой проект (вызов первичного конструктора).

```kotlin
class Car(val model: String, val year: Int) {

    // Вторичный конструктор. Принимает только модель (без года).
    // Через : this(...) он обязан передать данные в первичный конструктор, 
    // подставив год по умолчанию.
    constructor(model: String) : this(model, 2024) {
        println("Выпущен новый автомобиль базового года")
    }
}

```

### 2. Отказ от явных get/set и работа с `field` (Backing Field)

В Java разработчики вынуждены создавать приватное поле (например, `private String name;`) и писать к нему публичные методы `getName()` и `setName()`. Это нужно для инкапсуляции — чтобы контролировать доступ к данным.

В Kotlin **свойство — это уже готовая комбинация поля и его геттера/сеттера**. Когда ты пишешь `user.name`, ты не обращаешься к памяти напрямую, ты невидимо вызываешь геттер. Если стандартное поведение тебя устраивает, код остается абсолютно чистым.

Если же тебе нужно добавить логику (например, фильтровать мат в имени), ты можешь переопределить `get` и `set`. И здесь появляется ключевое слово **`field` (теневое поле)**.

* **Аналогия из жизни:** Банковская ячейка. Ты не можешь просто зайти в хранилище и взять деньги напрямую (прямой доступ к памяти). Ты подходишь к кассиру (геттер/сеттер). Кассир может проверить твой паспорт (дополнительная логика внутри `get()`), и только потом он оборачивается, открывает сейф у себя за спиной (это и есть `field`) и выдает тебе деньги.

```kotlin
class BankAccount {
    var balance: Int = 0
        // Переопределяем сеттер (set). 
        // value - это то значение, которое пытаются присвоить (например, account.balance = 500)
        set(value) {
            if (value >= 0) {
                // field - это магическое слово. Это "сейф" за спиной свойства.
                // Если мы напишем здесь balance = value, 
                // это вызовет сеттер внутри сеттера -> бесконечный цикл -> краш программы.
                // Поэтому мы кладем значение именно в скрытое поле 'field'.
                field = value 
            } else {
                println("Ошибка: Баланс не может быть отрицательным!")
            }
        }
        // Геттер (get) можно тоже переопределить, например, для логирования:
        get() {
            println("Кто-то запросил баланс")
            return field
        }
}

```

### 3. Отложенная инициализация: `lateinit var`

Вспомним прошлый модуль: Kotlin заставляет нас защищаться от `null`. Если мы объявили переменную как Non-Nullable (например, `var token: String`), мы **обязаны** дать ей значение прямо в момент создания класса.

Но что делать в мире энтерпрайза (например, в Spring Boot или при разработке под Android)? Очень часто классы создаются фреймворком автоматически, а зависимости (базы данных, токены, сервисы) "впрыскиваются" (Dependency Injection) чуть позже. Мы не можем инициализировать их сразу в конструкторе, но мы точно знаем, что к моменту реальной работы они там будут. Делать их Nullable (`String?`) — значит обречь себя на написание проверок `?.` или `!!` по всему коду.

Для этого существует **`lateinit var`** (отложенная инициализация). Ты обещаешь компилятору: *"Я клянусь, что положу сюда реальное значение до того, как кто-либо попытается его прочитать. Не заставляй меня делать его Nullable"*.

* **Аналогия из жизни:** Бронь гостиничного номера. Номер точно твой (он не `null`, это реальная физическая комната), но ты туда еще не заехал. Если кто-то (служба доставки) попытается принести туда пиццу до твоего заезда и постучит в дверь (попытка прочитать свойство) — он получит ошибку по лбу, так как номер еще "не инициализирован" жильцом.

```kotlin
class HotelRoom {
    // Мы не можем задать гостя сразу при создании комнаты, он заедет позже.
    // Но мы не хотим делать тип Guest?, потому что пустая комната нам не интересна в бизнес-логике.
    // Используем lateinit. (Работает только с var и только с объектами, не примитивами).
    lateinit var guestName: String

    fun checkIn(name: String) {
        guestName = name // Инициализация произошла
        println("$guestName заехал в номер.")
    }

    fun deliverPizza() {
        // Можно проверить, инициализирована ли переменная, чтобы избежать краша:
        if (this::guestName.isInitialized) {
            println("Пицца доставлена для $guestName")
        } else {
            println("Курьер ушел: номер пока пуст!")
        }
    }
}

fun main() {
    val room = HotelRoom()
    
    // Если раскомментировать строку ниже ДО вызова checkIn(), 
    // программа упадет с ошибкой UninitializedPropertyAccessException
    // println(room.guestName) 
    
    room.checkIn("Алексей")
    room.deliverPizza() // Теперь всё работает безопасно, никаких проверок на null не нужно!
}

```

---

Итог:

* Первичный конструктор (Primary) описывается в заголовке класса и позволяет в одну строчку объявить свойства (`val`/`var`) и присвоить им значения.
* Вторичные конструкторы (Secondary) позволяют создавать объекты альтернативными способами, но всегда обязаны сводиться к вызову первичного.
* В Kotlin нет необходимости вручную писать геттеры и сеттеры. Они генерируются компилятором. При необходимости их кастомизации используется скрытое поле `field`, чтобы избежать бесконечной рекурсии (StackOverflow).
* `lateinit var` решает проблему Dependency Injection (внедрения зависимостей): позволяет объявить свойство без `null`, отложив его фактическое заполнение на этап после создания объекта. Если обратиться к нему до заполнения — будет выброшено исключение `UninitializedPropertyAccessException`.
