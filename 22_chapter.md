## Глава 22. Продвинутые Generics

### 1. Проблема: Дженерики упрямы (Инвариантность)

В обычном ООП, если класс `Cat` (Кот) наследуется от класса `Animal` (Животное), мы можем положить кота в переменную типа животное. Это логично: кот — это животное.
Но с дженериками эта логика ломается. **Коробка с котами НЕ является коробкой с животными** для компилятора.

Почему? Это защита памяти!

```kotlin
open class Animal
class Cat : Animal()
class Dog : Animal()

// Обычный класс-коробка
class Box<T>(var item: T)

fun main() {
    val catBox: Box<Cat> = Box(Cat())
    
    // ОШИБКА КОМПИЛЯЦИИ! Компилятор запрещает это действие. Почему?
    // val animalBox: Box<Animal> = catBox 
    
    // Если бы компилятор это разрешил, мы могли бы сделать так:
    // animalBox.item = Dog() // Кладем собаку в коробку для животных.
    // И тогда в оригинальной catBox (коробке строго для котов) внезапно оказалась бы собака!
    // Программа упала бы с ошибкой памяти.
}

```

Такое жесткое поведение называется **инвариантностью** (ни шагу назад, ни шагу вперед). Но в реальной жизни нам нужна гибкость.

### 2. Ковариантность (Слово `out` / Продюсер)

Что, если мы пообещаем компилятору: *"Я буду **только доставать** вещи из этой коробки, я клянусь, что ничего не буду туда класть"*? Тогда угроза положить собаку в коробку с котами исчезает!

Чтобы дать компилятору такое обещание, мы ставим перед дженериком слово **`out`** (наружу).

* **Аналогия из жизни:** Фермерский рынок. Фермер продает только Яблоки. Ты ищешь поставщика любых Фруктов. Подойдет ли тебе поставщик Яблок? Да, идеально! Потому что ты будешь только *забирать (out)* у него товар. Ты же не собираешься привозить ему на ферму свои бананы (класть внутрь). Поэтому "Поставщик Яблок" является "Поставщиком Фруктов".

```kotlin
open class Fruit
class Apple : Fruit()

// Слово out говорит: "Этот класс будет ТОЛЬКО отдавать данные типа T. 
// Тип T может быть только в возвращаемом значении функций".
class Producer<out T>(private val item: T) {
    
    // Это разрешено (тип T возвращается НАРУЖУ)
    fun get(): T = item 
    
    // ОШИБКА! Компилятор не даст написать функцию, которая принимает T ВНУТРЬ
    // fun set(newItem: T) { ... } 
}

fun main() {
    val appleProducer: Producer<Apple> = Producer(Apple())
    
    // МАГИЯ! Теперь компилятор разрешает присвоить класс-наследник в класс-родитель.
    // Это и есть ковариантность.
    val fruitProducer: Producer<Fruit> = appleProducer 
    
    // Мы можем безопасно получать фрукты (они гарантированно будут яблоками)
    val myFruit: Fruit = fruitProducer.get()
}

```

*(Справка: Именно поэтому стандартный `List<T>` в Kotlin является ковариантным: он объявлен как `interface List<out E>`. Ты можешь только читать из него, он неизменяемый. Поэтому `List<String>` легко присваивается в переменную `List<Any>` без ошибок).*

### 3. Контравариантность (Слово `in` / Потребитель)

Это обратная ситуация. Что, если нам нужна коробка, которая умеет **только принимать** вещи, и нам вообще не важно, что она выдает обратно (или она ничего не выдает)?

Мы ставим слово **`in`** (внутрь). Это позволяет нам делать вещи, которые на первый взгляд ломают мозг: мы можем присвоить коробку с *родительским* типом в переменную с *дочерним* типом!

* **Аналогия из жизни:** Уничтожитель бумаг (Шредер). У тебя в офисе есть мощный промышленный шредер, который уничтожает "Любой мусор" (`Any`). Тебе на стол нужен маленький шредер строго для "Секретных документов" (`SecretDoc`). Подойдет ли тебе промышленный? Да! Если шредер может переварить *вообще всё*, то он легко справится с секретным документом. Ты будешь только *кидать (in)* в него бумаги.

```kotlin
open class Document
class SecretDocument : Document()

// Слово in говорит: "Этот класс будет ТОЛЬКО принимать данные типа T.
// Тип T может быть только в аргументах функций".
class Shredder<in T> {
    
    // Разрешено (тип T заходит ВНУТРЬ)
    fun destroy(item: T) {
        println("Уничтожено!")
    }

    // ОШИБКА! Компилятор не даст написать функцию, возвращающую T
    // fun retrieve(): T { ... }
}

fun main() {
    // Создаем шредер для ЛЮБЫХ документов (родительский тип)
    val universalShredder: Shredder<Document> = Shredder()

    // МАГИЯ НАОБОРОТ! Мы присваиваем РОДИТЕЛЯ в переменную НАСЛЕДНИКА.
    // Это и есть контравариантность.
    val secretShredder: Shredder<SecretDocument> = universalShredder

    // Почему это безопасно? 
    // Потому что через secretShredder мы можем закинуть только SecretDocument.
    // А наш universalShredder готов принять абсолютно любой Document. Все живы и счастливы.
    secretShredder.destroy(SecretDocument())
}

```

### Главное правило для запоминания (PECS)

В Java это правило называется **PECS** (*Producer Extends, Consumer Super*).
В Kotlin оно звучит гораздо проще и изящнее, прямо как читается:

* Нужно только отдавать данные? Пиши **`out`** (Producer).
* Нужно только принимать данные? Пиши **`in`** (Consumer).
* Нужно и то, и другое (например, изменяемый `MutableList`)? Не пиши ничего, но будь готов к строгой инвариантности (типы должны совпадать на 100%).

---

Итог:

* По умолчанию дженерики **инвариантны**: тип `Box<Наследник>` не является подтипом `Box<Родитель>`. Это сделано для строгой защиты памяти от случайной записи несовместимых данных.
* **Ковариантность (`out`)** позволяет обойти это ограничение, если класс гарантирует, что будет только *отдавать* (возвращать) параметризованный тип, но не принимать его. Это позволяет безопасно приводить `Producer<Наследник>` к `Producer<Родитель>`.
* **Контравариантность (`in`)** — обратный процесс. Если класс гарантирует, что будет только *принимать* (потреблять) параметризованный тип в качестве аргументов, мы можем безопасно приводить `Consumer<Родитель>` к `Consumer<Наследник>`.
* Модификаторы `in` и `out` в Kotlin указываются прямо при объявлении класса (Declaration-site variance), что делает использование таких классов в дальнейшем коде чистым и лишенным уродливых Java-конструкций (вроде `? extends T`).</Наследник></Родитель></Родитель></Наследник></Родитель></Наследник>
