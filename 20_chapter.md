## Глава 20. Gradle Kotlin DSL (.kts)

### 1. Проблема исторического наследия: Groovy

Изначально скрипты для Gradle писались на языке Groovy (файлы назывались просто `build.gradle`). Groovy — это язык с **динамической типизацией**. Это значит, что компилятор не проверяет типы переменных и существование методов до того момента, пока код не начнет выполняться.

* **Аналогия из жизни:** Список покупок на салфетке (Groovy). Ты пишешь от руки: *"Купить малако"*. Ты отдаешь салфетку другу, он едет в магазин, час бродит между полками, понимает, что "малака" не существует, и только потом звонит тебе с ошибкой. В программировании это называется **ошибкой времени выполнения (Runtime Error)**. Ты узнаешь об опечатке в скрипте сборки только тогда, когда запустишь долгий процесс компиляции проекта, и он упадет на середине.

В скриптах Groovy нет полноценного автодополнения кода (Т9 для программистов). Разработчикам приходилось копировать куски кода из интернета и надеяться, что они сработают.

### 2. Решение: Статическая типизация Kotlin DSL

Файлы **Kotlin DSL** (они имеют расширение `build.gradle.kts`) решают эту проблему радикально. Kotlin — это язык со **статической типизацией**. Компилятор знает всё о каждом объекте еще до запуска программы.

* **Аналогия из жизни:** Оформление доставки в мобильном приложении супермаркета (Kotlin DSL). Ты начинаешь вводить *"мал..."*, и приложение само (автодополнение) предлагает тебе варианты: *"Молоко 3.2%", "Молоко 1.5%"*. Если ты попытаешься вписать несуществующий товар руками, поле мгновенно загорится красным еще до того, как ты нажмешь кнопку "Заказать". В программировании это называется **ошибкой этапа компиляции (Compile-time Error)**.

Переход на `.kts` дал разработчикам три суперсилы:

1. **Идеальное автодополнение (Autocomplete) в IDE:** Среда разработки (например, IntelliJ IDEA) сама подсказывает, какие настройки можно написать.
2. **Мгновенная подсветка ошибок:** Если ты опечатался в названии настройки, слово сразу подчеркнется красным.
3. **Безопасный рефакторинг:** Ты можешь переименовать версию библиотеки в одном месте, и IDE безопасно обновит ее по всему скрипту.

### 3. Как это выглядит под капотом? (Связь с прошлыми модулями)

Когда новички смотрят на скрипт сборки Gradle, им кажется, что это какой-то отдельный конфигурационный язык вроде XML или JSON. Но если ты внимательно изучил модули 4.2 и 4.3 (Создание DSL и Лямбды с ресивером), ты испытаешь "вау-эффект": **скрипт Gradle `.kts` — это просто обычный код на Kotlin!**

Давай посмотрим на типичный блок подключения библиотек:

```kotlin
// Файл: build.gradle.kts

// 1. Блок plugins
// Кажется, что это магия сборщика. 
// На самом деле plugins - это просто функция высшего порядка, 
// которая принимает лямбду с ресивером (PluginDependenciesSpec.() -> Unit)
plugins {
    // kotlin - это вызов функции внутри ресивера, а version - инфиксная функция!
    kotlin("jvm") version "1.9.0" 
}

// 2. Блок dependencies (Зависимости)
// Это тоже функция высшего порядка с ресивером DependencyHandler.() -> Unit
dependencies {
    // В Groovy писали так (динамическая строка без скобок): 
    // implementation 'org.jetbrains.kotlin:kotlin-stdlib:1.9.0'
    
    // В Kotlin DSL мы вызываем РЕАЛЬНУЮ функцию implementation()
    // и передаем ей строку как аргумент. Строгая проверка компилятора!
    implementation("org.jetbrains.kotlinx:kotlinx-coroutines-core:1.7.3")
    
    // Если мы напишем implementTation (с опечаткой), 
    // Kotlin мгновенно подчеркнет это красным, потому что такой функции в ресивере нет.
    testImplementation("org.junit.jupiter:junit-jupiter:5.10.0")
}

```

### 4. Выделение логики (Выразительность)

Так как скрипт `.kts` — это полноценный Kotlin, мы можем прямо внутри скрипта сборки создавать обычные переменные, писать циклы `for`, использовать `if-else` и выносить константы.

```kotlin
// Мы можем создать обычную переменную (val) прямо в скрипте сборки, 
// чтобы не дублировать версию в разных местах.
val ktorVersion = "2.3.5"

dependencies {
    // Используем стандартную интерполяцию строк Kotlin ($)
    implementation("io.ktor:ktor-server-core:$ktorVersion")
    implementation("io.ktor:ktor-server-netty:$ktorVersion")
}

// Мы можем использовать стандартную логику языка (например, Scope-функции)
// Настроим компилятор Java, чтобы он использовал 17 версию
java {
    // sourceCompatibility и targetCompatibility - это обычные свойства (var) объекта java
    sourceCompatibility = JavaVersion.VERSION_17
    targetCompatibility = JavaVersion.VERSION_17
}

```

Благодаря Kotlin DSL настройка архитектуры проекта перестала быть "шаманством" со словарем и превратилась в написание строгого, безопасного и предсказуемого кода, где среда разработки всегда готова подсказать следующий шаг.

---

Итог:

* Скрипты сборки Gradle на Kotlin имеют расширение `.kts` (Kotlin Script). Они пришли на смену устаревшим динамическим скриптам на языке Groovy.
* Главное преимущество `.kts` — **статическая типизация**. Компилятор знает все доступные функции и свойства сборщика на этапе написания кода.
* Статическая типизация обеспечивает разработчика идеальным автодополнением (autocomplete) в IDE, мгновенной подсветкой опечаток и возможностью безопасного рефакторинга.
* Синтаксис `build.gradle.kts` не является встроенной "магией". Это классический DSL на базе Kotlin (разобранный в модуле 4.3), построенный на функциях высшего порядка и лямбдах с ресивером.
* Так как скрипт является полноценным кодом на Kotlin, в нем можно использовать весь арсенал языка: создавать переменные, циклы, функции-расширения и использовать строковую интерполяцию для управления версиями библиотек.
