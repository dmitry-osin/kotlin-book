## Глава 36. Отказ от Exceptions: Result<T>

### 1. Проблема Exceptions (Скрытая угроза)

Исключение (`Exception`) изначально задумывалось для непредсказуемых, катастрофических сбоев программы (кончилась оперативная память, пропало физическое соединение с жестким диском).
Но разработчики стали использовать `Exceptions` для обычной **бизнес-логики**: "Пароль неверный" -> `throw WrongPasswordException()`.

Почему это плохо?

1. **Ложь в контракте (сигнатуре):** Когда ты видишь функцию `fun getUser(id: Int): User`, компилятор обещает тебе вернуть `User`. Ты не знаешь, бросит ли она ошибку. Тебе нужно лезть внутрь её кода, чтобы узнать, нужно ли оборачивать её в `try/catch`.
2. **Эффект взорвавшейся гранаты (Control Flow):** Оператор `throw` мгновенно прерывает нормальный поток выполнения кода. Он работает как `GOTO`, выкидывая программу неведомо куда (в ближайший блок `catch`, который может находиться за 10 слоев выше).
3. **Огромная потеря ресурсов:** Создание `Exception` — это очень тяжелая операция для JVM. Виртуальная машина должна собрать весь *Stack Trace* (историю того, кто кого вызывал по всей цепочке), даже если тебе нужно просто сказать "Юзер не найден".

* **Аналогия из жизни:** Работа курьера.
* **Обычные Exceptions:** Ты отправляешь курьера за посылкой. Если на складе обед, курьер не звонит тебе, он **взрывает грузовик** (`throw Exception`). Взрывная волна ломает окна в офисе, срабатывает сигнализация (`catch`), и только из обрывков документов ты понимаешь, что склад был закрыт.
* **Функциональный подход:** Курьер приезжает на склад. Если там обед, он спокойно возвращается к тебе и отдает запечатанную коробку. В коробке лежит либо посылка (`Success`), либо официальная записка с причиной отказа (`Failure`). Никаких взрывов, рабочий процесс не прерывается.



### 2. Запечатанная коробка: `Result<T>`

В Kotlin (начиная с версии 1.3) появился встроенный `value class Result<T>`. Это та самая "запечатанная коробка", которая инкапсулирует в себе **либо** успешный результат типа `T`, **либо** ошибку `Throwable`.

Теперь наша функция не врет. Она честно говорит: *"Я могу вернуть пользователя, а могу вернуть ошибку. Тебе придется открыть коробку и проверить"*.

```kotlin
// Честный контракт. Функция возвращает Result!
fun findUser(id: Int): Result<String> {
    return if (id == 1) {
        Result.success("Алексей") // Упаковываем успех
    } else {
        Result.failure(IllegalArgumentException("Пользователь не найден")) // Упаковываем ошибку
    }
}

```

### 3. Изящный перехват: `runCatching`

Переписывать все старые библиотеки и сетевые запросы, которые по привычке кидаются `Exceptions`, было бы больно. Поэтому в Kotlin придумали функцию-страховку — **`runCatching`**.

Она работает как невидимая сетка под акробатом: ты выполняешь внутри нее опасный код (который может бросить `throw`), а она аккуратно ловит этот взрыв и упаковывает его в красивую коробочку `Result`.

```kotlin
fun dangerousNetworkCall(id: Int): String {
    if (id < 0) throw RuntimeException("Сеть упала!")
    return "Данные_$id"
}

fun main() {
    // Вместо уродливого try-catch мы используем runCatching
    val result: Result<String> = runCatching { 
        dangerousNetworkCall(-1) 
    }
    
    // Теперь ошибка не убила программу, она надежно спрятана внутри переменной result
    println("Коробка: $result") 
    // Выведет: Failure(java.lang.RuntimeException: Сеть упала!)
}

```

### 4. Функциональный конвейер (Pipeline)

Самая главная суперсила `Result` — это то, как мы распаковываем коробку. Вместо написания проверок `if-else`, мы строим красивый функциональный конвейер обработки данных с помощью цепочек методов `onSuccess`, `onFailure`, `map` и `recover`.

* **Аналогия из жизни:** Сортировочная лента на заводе. Едет коробка (`Result`). Стоит первый робот (`onSuccess`), он реагирует **только** если внутри деталь, и красит её. Стоит второй робот (`onFailure`), он реагирует **только** если внутри брак, и записывает это в журнал. Если едет брак, первый робот просто стоит и ничего не делает, пропуская коробку дальше.

```kotlin
fun main() {
    runCatching { 
        // 1. Делаем опасный запрос (возвращает String)
        dangerousNetworkCall(1) 
    }
    .map { text ->
        // 2. map сработает ТОЛЬКО если запрос успешен. 
        // Мы берем текст и, например, считаем его длину. 
        // Теперь коробка стала Result<Int>
        text.length 
    }
    .onSuccess { length ->
        // 3. Выполнится ТОЛЬКО при успехе
        println("Ура! Длина данных: $length") 
    }
    .onFailure { error ->
        // 4. Выполнится ТОЛЬКО при ошибке (в любом месте цепочки!)
        println("Упс, что-то пошло не так: ${error.message}") 
    }
    .recover { error ->
        // 5. Выживание. Если была ошибка, мы можем ее "вылечить" 
        // и подставить дефолтное значение (например, 0), чтобы программа пошла дальше.
        0
    }
    
    // .fold() - это конец конвейера. Он требует описать оба сценария 
    // и вернуть финальное значение.
    /*
    val finalData = result.fold(
        onSuccess = { data -> "Успех: $data" },
        onFailure = { error -> "Провал: ${error.message}" }
    )
    */
}

```

Этот подход делает код абсолютно линейным и читаемым, как стихи. Вся бизнес-логика (счастливый путь) идет сверху вниз, а обработка ошибок элегантно сдвигается в конец цепочки, не разрывая поток выполнения.

---

Итог:

* Использование традиционного `try/catch` для управления логикой (Control Flow) считается плохой практикой из-за скрытых контрактов функций и высоких затрат JVM на создание Stack Trace.
* Класс **`Result<T>`** представляет собой функциональную альтернативу (монаду). Это запечатанный тип, содержащий либо успешный результат (`T`), либо ошибку (`Throwable`), возвращаемый как обычный объект без прерывания программы.
* Функция **`runCatching { ... }`** позволяет безопасно выполнить старый код, склонный к выбросу исключений, и автоматически инкапсулировать результат или пойманную ошибку в класс `Result`.
* Обработка объекта `Result` строится в виде функционального конвейера (Pipelines). Методы `onSuccess`, `onFailure`, `map` (для изменения данных при успехе) и `recover` (для лечения ошибки) позволяют писать линейный, декларативный и абсолютно безопасный код.
