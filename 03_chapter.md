## Глава 3. Null-безопасность и Nullable типы

### 1. Разделение мира на `Type` и `Type?`

В Kotlin система типов строго разделена на две параллельные вселенные:

* **Non-Nullable (Обычные типы):** `String`, `Int`, `Boolean`. В них **категорически нельзя** положить `null`.
* **Nullable (Типы со знаком вопроса):** `String?`, `Int?`, `Boolean?`. В них можно положить как реальное значение, так и `null`.
* **Аналогия из жизни:** Обычный тип (`String`) — это посылка, которую тебе передают лично в руки. Ты на 100% уверен, что держишь коробку. Тип с вопросом (`String?`) — это почтовый ящик. Открывая его, ты можешь найти там письмо (реальное значение), а можешь обнаружить, что там пусто (`null`). Компилятор не даст тебе читать письмо из почтового ящика, пока ты не проверишь, есть ли оно там.

```kotlin
fun main() {
    // 1. Обычный тип. Компилятор гарантирует, что здесь всегда есть строка.
    var guaranteedName: String = "Алексей"
    // guaranteedName = null // Ошибка! Компилятор даже не даст запустить такой код.

    // 2. Nullable тип. Мы явно разрешаем переменной быть пустой.
    var unknownName: String? = "Иван"
    unknownName = null // Ошибки нет, ящик пуст.
    
    // Если мы попробуем напрямую узнать длину неизвестного имени:
    // val length = unknownName.length // Ошибка! Компилятор бьет по рукам: "А вдруг там null?"
}

```

### 2. Инструменты для работы с пустотой: `?.`, `?:` и `!!`

Так как компилятор запрещает напрямую обращаться к свойствам `Type?`, нам нужны специальные операторы, чтобы безопасно извлечь данные.

#### Оператор безопасного вызова `?.` (Safe Call)

Этот оператор говорит: «Если значение есть — выполни действие. Если там `null` — ничего не делай и просто верни `null`».

* **Аналогия:** Звонок мастеру. Если мастер взял трубку (не null), ты просишь его починить кран. Если он не берет трубку (null), ты же не кричишь в пустой телефон свою просьбу? Ты просто кладешь трубку (возвращается null).

#### Элвис-оператор `?:` (Elvis Operator)

Назван так, потому что похож на смайлик с прической Элвиса Пресли. Он позволяет задать "план Б". Если слева от него `null`, он вернет то, что справа.

* **Аналогия:** Запасное колесо в машине. Если основное колесо пробито (null), мы ставим запасное (значение по умолчанию) и едем дальше.

#### Жесткий досмотр `!!` (Not-null Assertion)

Этот оператор заставляет компилятор замолчать. Он говорит: «Я мамой клянусь, что здесь точно не `null`. Просто сделай это». Если ты ошибешься и там окажется `null`, программа упадет с той самой ошибкой `NullPointerException`.

* **Аналогия:** Прыжок с парашютом без проверки рюкзака. Ты слепо веришь, что парашют уложили. Если да — ты приземлишься. Если рюкзак пуст (null) — катастрофа. Использовать `!!` в профессиональном коде считается дурным тоном в 99% случаев.

```kotlin
fun main() {
    var userCity: String? = null

    // 1. Используем безопасный вызов (?.)
    // cityLength станет типом Int? (потому что может вернуться null)
    val cityLength: Int? = userCity?.length 

    // 2. Используем Элвис-оператор (?:) для Плана Б
    // Если userCity не null, берем длину. Если null, подставляем 0.
    // Теперь lengthNotNull имеет строгий тип Int (без вопроса!).
    val lengthNotNull: Int = userCity?.length ?: 0

    // 3. Используем жесткий досмотр (!!)
    // val crash: Int = userCity!!.length // ПРИВЕДЕТ К КРАШУ ПРОГРАММЫ (NPE)!
}

```

### 3. Потеря производительности: Почему `Int?` гарантирует Boxing

Здесь мы возвращаемся к прошлому модулю (1.2) и связываем память с безопасностью. Мы знаем, что обычный `Int` компилятор старается превратить в быстрый примитив `int` на уровне JVM (как число на легком стикере).

Но что произойдет, если мы объявим `Int?`?

Примитивы (типа `int` в памяти компьютера) — это просто электрические сигналы в ячейках памяти (нули и единицы). Они **не умеют** быть пустыми. У примитива всегда есть какое-то значение (например, если ячейка пуста для нас, технически там лежит `0`). Примитив не может означать `null`.

Поэтому, как только ты ставишь знак вопроса (`Int?`, `Double?`, `Boolean?`), компилятору ничего не остается, кроме как выполнить **Boxing (упаковку)**. Он создает тяжелый объект-обертку (`java.lang.Integer`). Почему? Потому что переменная-объект — это ссылка. А ссылка легко может указывать «в никуда» (то есть быть `null`).

* **Аналогия:** У тебя есть электронное табло для счета на стадионе (примитив). Оно физически не может быть "пустым", на нем всегда горят какие-то цифры, даже если это "00". Чтобы иметь возможность сказать «счета вообще нет», тебе нужно выключить табло из розетки, снять его и повесить табличку «Табло отсутствует» (создать объект со ссылкой `null`).

```kotlin
fun main() {
    // Это быстрый примитив 'int'. Никаких лишних затрат памяти.
    val fastAge: Int = 30

    // Как только мы добавили '?', компилятор выполнил Boxing.
    // В памяти был создан тяжелый объект Integer.
    // Если таких переменных будут миллионы (например, в массиве), 
    // программа начнет потреблять значительно больше оперативной памяти.
    val nullableAge: Int? = 30 
}

```

Поэтому в Kotlin есть негласное правило архитектуры: делай поля `Nullable` (со знаком вопроса) только тогда, когда отсутствие данных — это действительно заложенная бизнес-логика, а не просто "на всякий случай". Это экономит память.

---

Итог:

* В Kotlin система типов разделена на те, что не могут хранить пустоту (`Type`), и те, что могут (`Type?`). Это позволяет выявлять ошибки на этапе написания кода, а не во время работы программы.
* Для безопасной работы с Nullable типами используются: оператор `?.` (выполнить, если не пусто), Элвис-оператор `?:` (задать значение по умолчанию, если пусто).
* Оператор `!!` принудительно распаковывает значение, игнорируя защиту компилятора, и приводит к падению программы, если внутри оказался `null`. Его следует избегать.
* Использование Nullable-чисел (`Int?`, `Double?`) гарантированно приводит к упаковке (Boxing) примитивов в тяжелые объекты, так как сами по себе примитивы в памяти компьютера не могут принимать значение `null`. Это нужно учитывать при оптимизации.
