## Глава 30. CoroutineContext и Dispatchers

### 1. Что такое CoroutineContext?

Корутина никогда не висит в вакууме. У каждой корутины есть свой `CoroutineContext` — это набор правил и характеристик, по которым она живет.

Под капотом контекст работает как умный словарь (Map). В нем лежат четыре главных элемента:

1. **Job** (Жизненный цикл) — позволяет отменить корутину.
2. **CoroutineName** — имя корутины (полезно для логов).
3. **CoroutineExceptionHandler** — правило обработки ошибок (что делать, если всё упало).
4. **CoroutineDispatcher** (Диспетчер) — менеджер пула потоков.

* **Аналогия из жизни:** Личное дело сотрудника корпорации. В папке (Контексте) лежит: имя сотрудника (`Name`), его контракт с возможностью увольнения (`Job`), инструкция на случай пожара (`ExceptionHandler`) и пропуск в конкретный отдел здания (`Dispatcher`).

### 2. Dispatchers (Диспетчеры пулов потоков)

Диспетчер — это тот самый менеджер, который решает, на каком "железе" (в каком потоке) будет выполняться твой код. В стандартной библиотеке Kotlin есть три главных пула потоков. Выбор правильного диспетчера — это 90% успеха производительности твоего приложения.

#### 🏗️ Dispatchers.Default (Математики / Заводской цех)

* **Для чего:** Интенсивные вычисления (CPU-bound). Парсинг гигантских JSON, сортировка списков из миллиона элементов, фильтрация изображений, шифрование.
* **Особенность пула потоков:** Количество потоков в этом пуле **строго равно количеству ядер твоего процессора** (но не меньше двух).
* **Почему так? (Аналогия):** Представь заводской цех, где есть ровно 8 станков (ядер процессора). Работа здесь требует 100% концентрации и физической силы. Если ты наймешь 100 рабочих на 8 станков, они будут просто толкаться локтями, мешать друг другу и тратить время на смену мест (Context Switch). Идеальная эффективность — 8 рабочих на 8 станков.

#### 🚚 Dispatchers.IO (Грузчики / Склад)

* **Для чего:** Ожидание данных (I/O-bound). Скачивание файлов из интернета, запросы к базе данных, чтение файлов с диска.
* **Особенность пула потоков:** Этот пул может создавать сотни потоков (по умолчанию до 64, но можно больше).
* **Почему так? (Аналогия):** Склад приема товаров. Грузчик делает запрос (заказывает фуру с товаром) и... **просто ждет**. Ждать не сложно, для этого не нужны мощные станки (ядра процессора). Пока один грузчик ждет свою фуру у ворот, мы можем нанять еще 60 грузчиков, чтобы они тоже стояли у других ворот и ждали свои фуры. Поэтому здесь можно иметь кучу потоков — большую часть времени они просто "спят", ожидая ответа от сети или диска.

#### 👔 Dispatchers.Main (Ресепшн / Главный поток)

* **Для чего:** Обновление пользовательского интерфейса (UI) и легкая логика. Используется в Android, JavaFX, Swing.
* **Особенность пула потоков:** Поток всего **один**.
* **Почему так? (Аналогия):** Это девушка-администратор на ресепшене. Она должна мгновенно улыбаться клиентам (отрисовывать анимации, реагировать на клики). Если ты попросишь её пойти на склад разгрузить фуру (сделать сетевой запрос) или посчитать годовой налог (сложная математика), ресепшн опустеет. Клиенты будут жать на звонок, а им никто не ответит (приложение зависнет — ANR: Application Not Responding). **Главное правило: Никогда не блокируй Main поток тяжелыми задачами.**

### 3. Магия `withContext` (Переключение без боли)

В классическом программировании (например, в Java с коллбэками или в RxJava) переключение между потоками выглядит как "лапша" из кода: ты делаешь запрос в фоновом потоке, а потом пишешь специальный блок, чтобы вернуть результат в главный поток.

В Kotlin корутинах код выглядит как абсолютно плоский, синхронный текст благодаря функции **`withContext`**. Она позволяет корутине "перепрыгнуть" в другой пул потоков, выполнить там работу и элегантно вернуться обратно.

* **Аналогия из жизни:** Администратор на ресепшене (`Main`) принимает заказ от клиента. Она не идет на склад сама. Она по рации просит грузчика (`withContext(Dispatchers.IO)`) принести товар. Администратор *замирает* (`suspend`), но не блокирует двери — она может обслуживать других клиентов. Когда грузчик по рации сообщает, что товар здесь, администратор "просыпается" и отдает товар клиенту.

```kotlin
// Функция suspend! Мы предполагаем, что её вызвали из Dispatchers.Main (из UI)
suspend fun showUserProfile(userId: String) {
    println("[${Thread.currentThread().name}] Показываем крутилку загрузки (UI)")

    // Прыгаем в пул для сетевых запросов. 
    // Корутина на Main "засыпает", но сам главный поток НЕ БЛОКИРУЕТСЯ!
    val userJson = withContext(Dispatchers.IO) {
        println("[${Thread.currentThread().name}] Качаем данные из интернета...")
        fetchDataFromNetwork(userId) // Долгая сетевая операция
    } 

    // Прыгаем в пул для тяжелых вычислений, чтобы распарсить огромный JSON
    val userObject = withContext(Dispatchers.Default) {
        println("[${Thread.currentThread().name}] Парсим сложный JSON...")
        parseComplexJson(userJson) // Тяжелая математика
    }

    // withContext закончился. Мы автоматически вернулись в Dispatchers.Main!
    println("[${Thread.currentThread().name}] Прячем крутилку, рисуем имя: ${userObject.name}")
}

```

**Что происходит в логах при запуске этого кода:**

1. `[main]` Показываем крутилку загрузки (UI)
2. `[DefaultDispatcher-worker-1]` Качаем данные из интернета...
3. `[DefaultDispatcher-worker-3]` Парсим сложный JSON...
4. `[main]` Прячем крутилку, рисуем имя: Алексей

### 4. Наследование контекста (Вложенность)

Если ты запускаешь новую корутину внутри другой корутины, она **автоматически наследует** контекст родителя. Тебе не нужно указывать диспетчер каждый раз.

```kotlin
// Если эта корутина запущена в Dispatchers.Default
launch(Dispatchers.Default) {
    
    // Вложенная корутина автоматически запустится тоже в Default!
    launch {
        println("Я унаследовала Default пул от родителя")
    }
}

```

---

Итог:

* **CoroutineContext** — это набор параметров корутины (карта), определяющий её поведение. Самый важный элемент контекста — Диспетчер.
* **Диспетчеры (Dispatchers)** управляют распределением корутин по пулам физических потоков (Threads).
* **`Dispatchers.Main`** — используется исключительно для работы с UI. Он однопоточный, его категорически запрещено блокировать тяжелыми вычислениями или сетевыми вызовами.
* **`Dispatchers.IO`** — оптимизирован для операций ввода-вывода (сеть, диск, БД), где потоки большую часть времени "ждут". Пул может содержать большое количество потоков.
* **`Dispatchers.Default`** — оптимизирован для ресурсоемких вычислений (CPU-bound). Количество потоков жестко ограничено количеством ядер процессора, чтобы избежать накладных расходов на переключение контекста (Context Switch).
* Функция **`withContext(Dispatcher)`** позволяет безопасно и синхронно на вид переключить выполнение блока кода в другой пул потоков, приостанавливая (suspend) текущую корутину до завершения работы в фоне.
