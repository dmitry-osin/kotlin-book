## Глава 40. Аннотации взаимодействия с байткодом

### 1. `@JvmStatic`: Убираем костыль `Companion`

В Kotlin нет статических методов (`static`). Вместо них мы используем `companion object` (объект-компаньон). Но под капотом JVM компилятор создает реальный вложенный класс с именем `Companion` и кладет методы туда.

* **Аналогия из жизни:** Кабинет директора. В Java ты можешь зайти к директору напрямую (`Class.method()`). В Kotlin перед кабинетом сидит секретарь (Companion). Чтобы поговорить с директором из Java, тебе придется сначала обратиться к секретарю: `Class.Companion.method()`.

Аннотация **`@JvmStatic`** приказывает компилятору продублировать этот метод и сделать его по-настоящему статическим в родительском классе (убрать секретаря).

```kotlin
// --- КОД НА KOTLIN ---
class Config {
    companion object {
        fun getVersion(): String = "1.0.0"
        
        @JvmStatic
        fun getApiKey(): String = "XYZ-123"
    }
}

```

```java
// --- КАК ЭТО ВЫГЛЯДИТ ИЗ JAVA ---
public class JavaMain {
    public static void main(String[] args) {
        // БЕЗ аннотации: уродливый вызов через инстанс Companion
        String version = Config.Companion.getVersion(); 
        
        // С АННОТАЦИЕЙ: красивый и привычный статический вызов!
        String apiKey = Config.getApiKey(); 
    }
}

```

### 2. `@JvmOverloads`: Эмуляция дефолтных аргументов

В Kotlin мы обожаем аргументы по умолчанию (`val x = 0`). Это избавляет нас от написания десятков перегруженных методов. Но Java не поддерживает дефолтные аргументы! Если Java-разработчик вызовет твою Kotlin-функцию, он будет обязан передать **все** аргументы руками.

* **Аналогия из жизни:** Заказ в ресторане. В Kotlin ты говоришь: "Мне комбо-обед" (по умолчанию идет кола и картошка). Java так не умеет, ее меню строгое. Ей нужны три разные строчки в меню: "Бургер", "Бургер + Картошка", "Бургер + Картошка + Кола".

Аннотация **`@JvmOverloads`** заставляет компилятор сгенерировать все возможные варианты (перегрузки) этой функции в байткоде.

```kotlin
// --- КОД НА KOTLIN ---
class Alert {
    // Просим сгенерировать перегрузки
    @JvmOverloads
    fun show(text: String, duration: Int = 3, isCritical: Boolean = false) {
        println("Alert: $text")
    }
}

```

```java
// --- КАК ЭТО ВЫГЛЯДИТ ИЗ JAVA ---
// Под капотом компилятор создал 3 разных метода:
Alert alert = new Alert();

alert.show("Привет"); // Java-разработчик счастлив, он вызвал короткий метод!
alert.show("Привет", 5);
alert.show("Взлом!", 10, true);

```

### 3. `@JvmField`: Разрушение инкапсуляции (Прямой доступ)

В Kotlin нет публичных "полей". Когда ты пишешь `val name = "Алексей"`, компилятор прячет само значение в приватное поле (Backing Field) и автоматически создает публичный метод-геттер `getName()`.

Обычно это отлично. Но иногда старые Java-библиотеки (или инструменты сериализации) используют жесткую рефлексию и требуют, чтобы поле было **публичным (public) и без геттеров/сеттеров**.

Аннотация **`@JvmField`** приказывает компилятору: *"Не создавай геттеры и сеттеры. Оставь поле голым и публичным"*.

* **Аналогия из жизни:** Банковская ячейка. Обычное свойство Kotlin — это когда ты просишь клерка (геттер) принести тебе содержимое ячейки. `@JvmField` — это когда ты выламываешь дверь хранилища и берешь деньги из ячейки напрямую своими руками.

```kotlin
// --- КОД НА KOTLIN ---
class User {
    val id: String = "123" // Будет сгенерирован метод getId()
    
    @JvmField
    val role: String = "Admin" // Будет просто public String role;
}

```

```java
// --- КАК ЭТО ВЫГЛЯДИТ ИЗ JAVA ---
User user = new User();
String myId = user.getId(); // Доступ через геттер
String myRole = user.role;  // Прямой доступ к полю!

```

### 4. `@JvmName`: Переименование в байткоде (Спасение от Type Erasure)

Мы помним из модуля 5.2 (Стирание типов), что в рантайме JVM не отличает `List<String>` от `List<Int>`. Из-за этого мы не можем написать две функции с одинаковым названием, но разными дженериками — компилятор выдаст ошибку конфликта сигнатур (*Platform declaration clash*).

Аннотация **`@JvmName`** позволяет изменить имя метода **только в скомпилированном Java-байткоде**, оставив его прежним для Kotlin-кода.

* **Аналогия из жизни:** Сценический псевдоним. По паспорту (в байткоде) певца зовут "Эдуард Сидоров", чтобы налоговая (JVM) его не перепутала с другим Эдуардом. Но на афише (в коде Kotlin) он всегда выступает как "Эдди".

```kotlin
// --- КОД НА KOTLIN ---

// ОШИБКА КОМПИЛЯЦИИ! JVM видит оба метода как printList(List list)
// fun printList(list: List<String>) { ... }
// fun printList(list: List<Int>) { ... }

// РЕШЕНИЕ:
fun printList(list: List<String>) { println("Строки") }

@JvmName("printIntList") // Меняем имя для JVM
fun printList(list: List<Int>) { println("Числа") }

fun main() {
    // В Kotlin мы вызываем их одинаково красиво!
    printList(listOf("A", "B"))
    printList(listOf(1, 2))
}

```

*(Также `@JvmName` часто используют над файлами: `@file:JvmName("Utils")`, чтобы глобальные функции из файла `UtilsKt.kt` в Java вызывались красиво: `Utils.doSomething()` вместо `UtilsKt.doSomething()`).*

### 5. `@Throws`: Возвращаем проверяемые исключения (Checked Exceptions)

В Java исключения делятся на проверяемые (Checked) и непроверяемые (Unchecked). Если Java-метод может бросить проверяемое исключение (например, `IOException` при чтении файла), Java **обязывает** программиста обернуть этот вызов в `try/catch`.

В Kotlin решили, что Checked Exceptions — это зло, поэтому **в Kotlin все исключения непроверяемые**. Компилятор Kotlin никогда не заставит тебя писать `try/catch`.

Но если твою Kotlin-функцию вызовут из Java, и она бросит `IOException`, Java-компилятор сойдет с ума. Он не позволит Java-разработчику написать `catch (IOException e)`, потому что по сигнатуре (в байткоде) этот метод ничего не бросает!

Аннотация **`@Throws`** добавляет в байткод официальную приписку `throws ExceptionName`, чтобы Java-компилятор успокоился.

* **Аналогия из жизни:** Предупреждающий знак на дороге. Kotlin-разработчики ездят без знаков, доверяя своей реакции. Но чтобы пустить на эту дорогу Java-разработчика (у которого строгие ПДД), нам нужно официально повесить знак `@Throws("Осторожно, обрыв")`, иначе его машина просто не заведется.

```kotlin
import java.io.IOException

// --- КОД НА KOTLIN ---
class FileReader {
    
    // Kotlin не требует try/catch, но мы предупреждаем Java-мир!
    @Throws(IOException::class)
    fun readFile(path: String) {
        if (path.isEmpty()) {
            throw IOException("Файл не найден")
        }
    }
}

```

```java
// --- КАК ЭТО ВЫГЛЯДИТ ИЗ JAVA ---
FileReader reader = new FileReader();

// Благодаря @Throws, Java-компилятор ОБЯЗЫВАЕТ нас (или разрешает нам) 
// написать этот блок catch. Без аннотации была бы ошибка компиляции в Java.
try {
    reader.readFile("test.txt");
} catch (IOException e) {
    System.out.println("Поймали ошибку: " + e.getMessage());
}

```

---

Итог:

* Аннотации `@Jvm...` — это инструменты для тонкой настройки генерации байткода. Они необходимы исключительно для обеспечения гладкой совместимости (Interoperability) Kotlin-кода с языком Java и старыми JVM-фреймворками.
* **`@JvmStatic`** заставляет компилятор генерировать реальный `public static` метод в родительском классе, избавляя Java-клиентов от необходимости вызывать методы через инстанс `Companion`.
* **`@JvmOverloads`** генерирует телескопические перегрузки методов для Java, эмулируя функционал аргументов по умолчанию из Kotlin.
* **`@JvmField`** отключает генерацию геттеров и сеттеров, выставляя свойство как публичное поле (public field) в байткоде (полезно для рефлексии и оптимизации производительности).
* **`@JvmName`** позволяет изменить физическое имя метода или класса в скомпилированном байткоде. Решает проблему конфликта сигнатур (Type Erasure) при использовании дженериков.
* **`@Throws`** добавляет секцию `throws` в сигнатуру метода на уровне JVM, позволяя (и обязывая) Java-потребителей корректно обрабатывать выбрасываемые "проверяемые" (checked) исключения.
