## Глава 7. Иерархия: open и abstract

### 1. Почему все классы закрыты по умолчанию (`final`)?

В Java любой класс по умолчанию открыт. Кто угодно может унаследовать твой класс, переопределить его методы и изменить его поведение. В Kotlin разработчики взяли за основу знаменитое правило из книги Джошуа Блоха «Effective Java»: *«Проектируйте для наследования или запрещайте его»*.

Поэтому в Kotlin **абсолютно все классы и их методы по умолчанию закрыты** (под капотом JVM они помечаются как `final`). От них нельзя наследоваться.

* **Аналогия из жизни:** Обычный смартфон или микроволновка. Производитель выпускает готовый продукт и наглухо запаивает корпус. Он гарантирует, что устройство работает идеально именно в такой сборке. Если бы корпус был открыт (как в Java), любой пользователь мог бы влезть туда с паяльником, заменить пару деталей и потом жаловаться, что микроволновка взорвалась. Kotlin "запаивает корпус", защищая внутреннюю логику класса от случайной поломки (это называется проблемой хрупкого базового класса — Fragile Base Class).

```kotlin
// Обычный класс. По умолчанию он "запаян" (final).
class Microwave {
    fun heatFood() {
        println("Грею еду 2 минуты")
    }
}

// ОШИБКА КОМПИЛЯЦИИ! 
// Мы не можем унаследовать "Умную микроволновку" от обычной, потому что класс закрыт.
// class SmartMicrowave : Microwave() 

```

### 2. Ключевое слово `open`: Разрешение на модификацию

Если ты как архитектор задумываешь класс как основу для расширения, ты должен **явно** разрешить это с помощью слова `open`. Причем разрешить нужно не только само наследование класса, но и переопределение конкретных методов внутри него.

* **Аналогия из жизни:** Системный блок настольного компьютера или модульный дом. Архитектор изначально спроектировал съемную боковую крышку и пустые слоты (open класс), чтобы ты мог добавить туда новую видеокарту или оперативную память (open методы).

```kotlin
// Слово open говорит: "Этот класс спроектирован для наследования, я за него ручаюсь".
open class BasicComputer {
    
    // Этот метод обычный (final). Наследники смогут его использовать, 
    // но НЕ СМОГУТ изменить его внутреннюю логику.
    fun turnOn() {
        println("Питание подано. Загрузка...")
    }

    // Слово open перед функцией разрешает наследникам переписать эту логику под себя.
    open fun calculate() {
        println("Выполняю базовые вычисления")
    }
}

// Класс-наследник. Через двоеточие мы указываем родителя и вызываем его конструктор ().
class GamingComputer : BasicComputer() {
    
    // Мы обязаны использовать слово override, чтобы показать, 
    // что мы осознанно меняем родительскую функцию, а не случайно назвали ее так же.
    override fun calculate() {
        // Мы можем вызвать оригинальную логику через super (если нужно)
        super.calculate() 
        println("Запускаю рендеринг графики на мощной видеокарте!")
    }
}

```

### 3. Ключевое слово `abstract`: Обязательный шаблон

Иногда нам нужна еще более строгая иерархия. `abstract` класс — это концепция, некий общий чертеж. Создать реальный объект (экземпляр) абстрактного класса физически невозможно. Он существует только для того, чтобы от него наследовались другие.

В отличие от `open` класса, где методы уже имеют какую-то реализацию, абстрактный класс может содержать "пустые" методы (без тела). Наследники **обязаны** их реализовать.

* **Аналогия из жизни:** Чертеж транспортного средства в конструкторском бюро. Ты не можешь спуститься в цех и попросить рабочих собрать просто «Транспортное средство». Это абстракция. Но на основе этого чертежа можно собрать «Легковой автомобиль» или «Мотоцикл». При этом в абстрактном чертеже сказано: *«У любого транспорта должен быть метод 'ехать', но как именно он будет крутить колеса — решает конкретный наследник»*.

```kotlin
// Создать объект этого класса нельзя: val t = Vehicle() выдаст ошибку.
abstract class Vehicle(val name: String) {
    
    // Обычный метод с реализацией. Доступен всем наследникам.
    // (Абстрактные классы автоматически являются open, писать open class не нужно).
    fun startEngine() {
        println("$name: Двигатель запущен.")
    }

    // Абстрактный метод. У него нет фигурных скобок и тела.
    // Это просто жесткое требование: "Любой наследник ОБЯЗАН это уметь".
    abstract fun drive()
}

// Класс-наследник
class Car(name: String) : Vehicle(name) {
    
    // Компилятор бьет по рукам, пока мы не реализуем абстрактный метод родителя.
    override fun drive() {
        println("$name: Еду на четырех колесах, жгу бензин.")
    }
}

class Bicycle(name: String) : Vehicle(name) {
    
    override fun drive() {
        println("$name: Еду на двух колесах, кручу педали.")
    }
}

```

---

Итог:

* В Kotlin все классы и их методы по умолчанию являются закрытыми (`final`). Это архитектурное решение защищает код от непредсказуемого изменения логики базовых классов (проблема Fragile Base Class).
* Чтобы класс можно было унаследовать, его необходимо явно пометить модификатором `open`. То же самое касается методов: чтобы их можно было переопределить в наследниках (через `override`), они тоже должны быть `open`.
* Модификатор `abstract` используется для создания "классов-чертежей". Невозможно создать экземпляр абстрактного класса.
* Абстрактные классы могут содержать абстрактные методы (без реализации). Классы-наследники берут на себя строгую обязанность предоставить конкретную реализацию для каждого такого метода.
