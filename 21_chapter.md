## Глава 21. Contracts и Context Receivers

### 1. Проблема потери контекста (Глупый компилятор)

Ты уже знаешь, что в Kotlin есть **Умное приведение типов (Smart Cast)**. Если ты проверил переменную на `null`, компилятор запоминает это и разрешает работать с ней как с безопасной (без знака вопроса).

Но компилятор "умный" только в пределах одной функции. Как только ты выносишь логику проверки в отдельный метод, компилятор внезапно "глупеет". Он не заглядывает внутрь других функций.

* **Аналогия из жизни:** Охранник и Бармен. Охранник на входе (твоя функция проверки) проверяет паспорт и говорит: "Парню есть 18". Но когда парень подходит к бармену (компилятору), бармен всё равно требует паспорт заново, потому что он не видел сам процесс проверки и не доверяет словам охранника.

```kotlin
// Наша удобная функция для проверки
fun isReady(text: String?): Boolean {
    return text != null && text.isNotBlank()
}

fun processData(data: String?) {
    // Вызываем нашу проверку
    if (isReady(data)) {
        // ОШИБКА КОМПИЛЯЦИИ! 
        // Компилятор не знает, что внутри isReady мы уже проверили на null.
        // Он заставляет нас писать data?.length или data!!.length
        // println("Длина: ${data.length}") 
    }
}

```

### 2. Contracts (Контракты): Юридический договор с компилятором

Чтобы решить эту проблему, разработчики Kotlin придумали **Contracts (Контракты)**. Это способ написать для компилятора строгую юридическую гарантию: *"Я клянусь, что если эта функция вернет `true`, то переданная переменная точно не `null`"*.

Для этого в самом начале функции открывается специальный блок `contract { ... }`.

* **Аналогия из жизни:** Сертифицированный инспектор. Теперь охранник не просто говорит "ему есть 18". Он ставит официальную печать с голограммой (Контракт) на руку парня. Бармен (компилятор) видит эту печать, юридически доверяет ей и наливает напиток без лишних проверок.

```kotlin
import kotlin.contracts.ExperimentalContracts
import kotlin.contracts.contract

// (Пока создание своих контрактов требует специального разрешения, так как фича мощная)
@OptIn(ExperimentalContracts::class)
fun isValidString(text: String?): Boolean {
    // Заключаем контракт с компилятором в самом начале функции
    contract {
        // Читается как текст: "Если функция возвращает true, 
        // это ПОДРАЗУМЕВАЕТ (implies), что text не равен null"
        returns(true) implies (text != null)
    }
    
    return text != null && text.isNotBlank()
}

fun processSafeData(data: String?) {
    if (isValidString(data)) {
        // МАГИЯ! Компилятор прочитал контракт.
        // Smart Cast сработал! Мы обращаемся к data напрямую, без '?'
        println("Строка валидна. Длина: ${data.length}")
    }
}

```

*Кстати, многие стандартные функции Kotlin (например, `isNullOrEmpty()` или `require()`) под капотом написаны именно с использованием контрактов!*

### 3. Context Receivers: Инъекция контекста без боли

В модулях 3.2 и 4.2 мы изучили функции-расширения и ресиверы (`Type.() -> Unit`). Они прекрасны, но у них есть одно жесткое ограничение: **у функции может быть только один Receiver (один `this`)**.

Но в реальном (Enterprise) программировании функции часто требуют **сразу несколько контекстов** для работы. Например, чтобы сохранить платеж, нам нужен контекст `Database` (чтобы писать в базу) и контекст `Logger` (чтобы писать логи). Передавать их каждый раз в аргументах функции — значит засорять код.

**Context Receivers** (Контекстные получатели) — это механизм, который говорит: *"Эту функцию можно вызвать ТОЛЬКО если в текущем месте одновременно доступны и База Данных, и Логгер"*.

* **Аналогия из жизни:** Хирург в операционной. Чтобы провести сложную операцию (выполнить функцию), хирургу недостаточно просто пациента (аргумент). Ему абсолютно необходимы два контекста: «Стерильная операционная» и «Бригада анестезиологов». Хирург не носит операционную с собой в кармане (в аргументах). Он просто знает, что когда он зайдет в нужную комнату (зону видимости), эти вещи уже будут там, и он сможет ими воспользоваться.

```kotlin
// Создаем два независимых класса (контекста)
class Logger {
    fun log(msg: String) = println("[LOG]: $msg")
}

class Database {
    fun save(data: String) = println("[DB]: Данные '$data' успешно сохранены")
}

// Используем ключевое слово context(...)
// Это значит: Функция processPayment понятия не имеет, ОТКУДА возьмутся Logger и Database,
// но она требует, чтобы при ее вызове они "витали в воздухе" (были в области видимости).
context(Logger, Database)
fun processPayment(amount: Double) {
    // Внутри функции мы имеем прямой доступ к методам ОБОИХ контекстов!
    log("Начало обработки платежа на сумму $amount") // Вызов метода из Logger
    save("Платеж_$amount")                           // Вызов метода из Database
    log("Платеж завершен")
}

fun main() {
    val myLogger = Logger()
    val myDb = Database()

    // processPayment(500.0) // ОШИБКА! В воздухе (в main) нет контекстов логгера и базы.

    // Создаем нужную среду. 
    // С помощью 'with' мы "заходим в комнату", где лежат myLogger и myDb.
    with(myLogger) {
        with(myDb) {
            // Только здесь, внутри двух контекстов, функция "оживает" и позволяет себя вызвать.
            // При этом мы не передаем логгер и базу внутрь скобок!
            processPayment(500.0) 
        }
    }
}

```

*(Примечание: Дизайн передачи контекстов в Kotlin сейчас активно эволюционирует, и в будущих версиях языка ключевое слово `context` может быть заменено на более удобные `context parameters` (параметры контекста), но концепция инъекции окружения "по воздуху", без захламления аргументов, останется неизменной).*

---

Итог:

* **Contracts (Контракты)** — это механизм, позволяющий разработчику "объяснить" компилятору неочевидную логику работы функции.
* Контракты чаще всего используются для обучения умного приведения типов (Smart Cast). С помощью конструкции `returns(true) implies (value != null)` компилятор начинает понимать, что после успешного выполнения твоей функции переменную можно безопасно использовать без проверок на `null`.
* **Context Receivers** решают проблему нехватки одного `this` в функциях-расширениях. Они позволяют требовать наличия сразу нескольких независимых контекстов (объектов) для выполнения функции.
* Использование `context(A, B)` позволяет функции прозрачно использовать методы классов `A` и `B`, не принимая их в качестве прямых аргументов. Это мощнейший инструмент для чистой инъекции зависимостей (Dependency Injection) на уровне функций.
