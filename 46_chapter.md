## Глава 46. Kotlin Multiplatform (KMP)

### 1. Проблема: Ужасы кроссплатформенности

Исторически у IT-компаний было два пути для разработки клиентских приложений, и оба причиняли боль:

1. **Нативная разработка (iOS + Android отдельно):** Компания нанимает две команды. Пишутся два приложения с нуля: одно на Swift, другое на Kotlin. Бизнес-логика (парсинг JSON, кеширование, валидация) дублируется на 100%. Ошибки тоже дублируются. Это дорого и медленно.
2. **Кроссплатформенные фреймворки (Flutter / React Native):** Пишется одно приложение. Но эти фреймворки забирают контроль над **всем**, включая пользовательский интерфейс (UI). В итоге приложение на iPhone выглядит "не совсем как iPhone", ломаются стандартные жесты, а производительность страдает из-за тяжелого моста между JavaScript/Dart и нативной системой.

### 2. Философия KMP: Делись мозгами, а не лицом

Создатели Kotlin выбрали совершенно другой, прагматичный архитектурный подход.

Девиз KMP: **Шаринг бизнес-логики при сохранении нативного UI.**

В любом приложении 70% кода — это "невидимая" математика: запросы в сеть, сохранение в базу данных (SQL), фильтрация списков, валидация паролей. И только 30% — это отрисовка кнопок и анимаций.
KMP позволяет вынести эти 70% в общий модуль (`commonMain`).

* **Аналогия из жизни:** Международная сеть ресторанов.
* **KMP (Бизнес-логика):** Это централизованная кухня, рецептуры, логистика и расчет налогов. Они абсолютно одинаковые и в Москве, и в Токио.
* **Нативный UI (iOS / Android):** Это интерьер зала, униформа официантов и язык меню. В Токио зал выглядит по-японски, в Москве — по-европейски. Рестораны делят общую эффективную "базу", но для клиента выглядят полностью "своими".



### 3. Механика взаимодействия: `expect` / `actual`

Если общий код (`commonMain`) ничего не знает о платформе, как нам получить доступ к вещам, которые работают везде по-разному? Например, как узнать версию операционной системы или получить UUID устройства?

Для этого в KMP придуман элегантный механизм — ключевые слова **`expect`** (ожидаю) и **`actual`** (реализую).

* **Аналогия из жизни:** Директор дает поручение. Директор (`expect` в общем коде) говорит: *"Мне нужен отчет о продажах. Я не знаю, как вы его сделаете, но к вечеру он должен лежать у меня на столе"*.
Менеджер отдела iOS (`actual`) делает его в Excel на маке. Менеджер отдела Android (`actual`) пишет его от руки в блокноте. Директору не важен процесс, ему важен гарантированный результат.

```kotlin
// --- ОБЩИЙ КОД (commonMain) ---
// Мы просто объявляем, что такая функция ДОЛЖНА быть.
// Самого кода здесь нет.
expect fun getDeviceUUID(): String 

fun logUserAction() {
    // Мы спокойно используем эту функцию в общей бизнес-логике!
    println("Действие от: ${getDeviceUUID()}")
}

// --- КОД ДЛЯ ANDROID (androidMain) ---
import java.util.UUID

// Компилятор заставит нас написать actual-реализацию для Android
actual fun getDeviceUUID(): String {
    return UUID.randomUUID().toString() // Используем Java API
}

// --- КОД ДЛЯ iOS (iosMain) ---
import platform.Foundation.NSUUID

// И actual-реализацию для iOS
actual fun getDeviceUUID(): String {
    return NSUUID().UUIDString // Используем Apple iOS API!
}

```

### 4. Куда компилируется Kotlin? (Targets)

Магия KMP заключается в том, что один и тот же язык Kotlin компилируется в совершенно разные бинарные форматы с помощью разных бэкендов компилятора:

* **JVM / Android (`Kotlin/JVM`):** Код превращается в стандартные `.class` / `.dex` файлы. Для Android-приложения (или Spring-бэкенда) твой KMP-модуль выглядит как обычная Java-библиотека.
* **iOS / Desktop (`Kotlin/Native`):** Здесь нет никакой виртуальной машины (JVM). Код компилируется напрямую в машинный код (через инфраструктуру LLVM) и упаковывается в `.framework`. Swift-разработчик просто импортирует его в XCode и вызывает методы так, словно они написаны на Objective-C!
* **WebAssembly (`Kotlin/Wasm`):** Новейший и самый амбициозный рубеж. Код компилируется в `.wasm` бинарник. WebAssembly работает прямо внутри браузера (Chrome, Safari), выполняя сложную математику (например, парсинг гигантских JSON или криптографию) на около-нативной скорости, недоступной обычному JavaScript.

### 5. Compose Multiplatform: Следующий шаг

Хотя изначальная идея была в шаринге только бизнес-логики, JetBrains пошли дальше. Опираясь на успех Jetpack Compose (декларативного UI для Android), они создали **Compose Multiplatform (CMP)**.

Теперь, если проект позволяет (например, это внутреннее B2B-приложение или админка), ты можешь писать на Kotlin даже пользовательский интерфейс, и он сам отрисуется на iOS (через Skia/Metal), на Десктопе (Windows/Mac/Linux) и в браузере.
