## Глава 37. Библиотека kotlinx.serialization

### 1. Проблема Gson и Jackson: Проклятие Рефлексии

Мы изучали Рефлексию (Reflection) в модуле 5.6. Это способность программы во время работы (в рантайме) сканировать саму себя, чтобы узнать, какие классы и свойства в ней существуют.

Именно так работают Gson и Jackson. Когда к ним приходит строка `{"name": "Алексей"}`, они берут класс `User`, с помощью тяжелой рефлексии "ощупывают" его, находят свойство `name` и с помощью динамических вызовов пытаются записать туда значение.

**Почему это плохо?**

1. **Это невероятно медленно.** Анализ классов в рантайме отнимает ценные такты процессора.
2. **Это небезопасно для Kotlin.** Рефлексия Java ничего не знает о правилах Kotlin (например, о Nullable типах `?` или значениях по умолчанию). Gson может спокойно подсунуть `null` в переменную `val name: String` (которая строго не Nullable), и программа упадет с `NullPointerException` не в момент парсинга, а позже, в самом неожиданном месте бизнес-логики.
3. **Несовместимость с мобильной разработкой.** Android (с инструментом R8/ProGuard) и Kotlin Native (iOS/Desktop) активно удаляют "неиспользуемый" код для уменьшения размера приложения. Они могут удалить имена твоих свойств, и тогда библиотека на основе рефлексии просто не сможет найти, куда положить данные (придется писать длинные списки правил исключений).

### 2. Решение: `kotlinx.serialization` (Генерация кода компилятором)

Разработчики Kotlin создали гениальный инструмент, который работает **без рефлексии**.

* **Аналогия из жизни:**
* **Gson (Рефлексия):** Сборка мебели вслепую. Сборщик пришел к тебе домой без инструкции. Он берет каждую доску, ощупывает ее (Рефлексия), пытается приложить к другим доскам, чтобы понять, что куда подходит. Это долго и чревато ошибками.
* **kotlinx.serialization:** Заводская инструкция. Еще на заводе (на этапе компиляции кода) умная машина печатает подробную, жесткую инструкцию по сборке (сгенерированный код). Когда сборщик приходит к тебе, он не думает и не щупает — он просто берет деталь А и вставляет в паз Б за секунду.



Когда ты помечаешь класс аннотацией `@Serializable`, **компилятор Kotlin плагин** на этапе сборки программы физически пишет за тебя отдельный скрытый класс — "Сериализатор". Этот класс жестко, прямым кодом (без всякой рефлексии) знает, как разобрать именно твой класс `User` по запчастям и собрать обратно.

### 3. Как это выглядит в коде

Для начала нужно подключить плагин компилятора в `build.gradle.kts` и добавить саму библиотеку.
*(Напоминание из модуля 4.4: Gradle скрипт — это просто Kotlin код!)*

```kotlin
plugins {
    kotlin("jvm") version "1.9.0"
    // Подключаем плагин для генерации кода!
    kotlin("plugin.serialization") version "1.9.0" 
}
dependencies {
    implementation("org.jetbrains.kotlinx:kotlinx-serialization-json:1.6.0")
}

```

Теперь перейдем к коду. Все, что нам нужно — это добавить одну аннотацию.

```kotlin
import kotlinx.serialization.*
import kotlinx.serialization.json.*

// Волшебная аннотация. Компилятор увидит её и сгенерирует 
// весь необходимый, сверхбыстрый код под капотом.
@Serializable
data class User(
    val name: String,
    val age: Int,
    // Если в JSON может не быть этого поля, задаем дефолтное значение
    val role: String = "guest" 
)

fun main() {
    val myUser = User("Дмитрий", 30)

    // 1. Сериализация (Объект -> JSON строка)
    val jsonString = Json.encodeToString(myUser)
    println("JSON: $jsonString") 
    // Вывод: JSON: {"name":"Дмитрий","age":30,"role":"guest"}

    // 2. Десериализация (JSON строка -> Объект)
    // Используем reified (из модуля 5.2!), чтобы не передавать User::class.java
    val rawJson = """{"name": "Анна", "age": 25}"""
    val decodedUser = Json.decodeFromString<User>(rawJson)
    
    println("Объект: $decodedUser")
    // Вывод: Объект: User(name=Анна, age=25, role=guest)
    // Магия! Библиотека поняла, что поля role в JSON нет, 
    // и корректно подставила "guest", не сломав строгие типы Kotlin!
}

```

### 4. Настройка JSON и `@SerialName`

Часто имена свойств в твоем коде (например, *camelCase* `firstName`) не совпадают с тем, как их принято называть на сервере (например, *snake_case* `first_name`).

Чтобы подружить их, не меняя красивый код на Kotlin, используется аннотация `@SerialName`.

```kotlin
@Serializable
data class Response(
    // В Kotlin мы хотим писать id, 
    // но сервер почему-то присылает ключ "user_identifier"
    @SerialName("user_identifier")
    val id: Int,
    
    val status: String
)

```

**Строгость по умолчанию:**
`kotlinx.serialization` невероятно строгая по умолчанию. Если сервер пришлет тебе JSON с ключом `"unknown_field"`, которого нет в твоем классе `Response`, парсер немедленно упадет с ошибкой `UnknownFieldException`.
Это сделано для безопасности. Но если ты не контролируешь сервер и там может быть "мусор", ты можешь настроить экземпляр класса `Json`:

```kotlin
// Создаем свой, настроенный экземпляр парсера
val relaxedJson = Json { 
    ignoreUnknownKeys = true // Игнорировать лишние поля (не падать)
    prettyPrint = true       // Делать красивый формат с отступами (для логов)
    encodeDefaults = true    // Записывать в JSON поля даже с дефолтными значениями
}

val jsonString = relaxedJson.encodeToString(User("Алексей", 20))

```

Благодаря тому, что `kotlinx.serialization` работает на этапе компиляции, она является самой быстрой библиотекой для Kotlin, поддерживает мультиплатформенность (KMP) и гарантирует абсолютную типобезопасность.

---

Итог:

* Классические библиотеки сериализации (Gson, Jackson) используют **рефлексию** (runtime analysis). Это приводит к снижению производительности, конфликтам с обфускаторами (R8/ProGuard) и потенциальным нарушениям Null-безопасности Kotlin.
* **`kotlinx.serialization`** — официальная библиотека JetBrains, работающая на базе плагина компилятора.
* При наличии аннотации `@Serializable` компилятор генерирует специальный код сериализатора для конкретного класса на этапе сборки. Это исключает необходимость использования рефлексии (Zero-reflection), делая процесс сверхбыстрым и мультиплатформенным (совместимым с Kotlin/Native и Kotlin/JS).
* Библиотека полностью интегрирована с системой типов Kotlin: она уважает Nullable-типы, корректно применяет значения по умолчанию и не позволяет обойти строгую типизацию.
* Для маппинга полей (при несовпадении имен в JSON и классе) применяется аннотация `@SerialName`. Поведение парсера настраивается через переопределение экземпляра `Json { ... }` (например, `ignoreUnknownKeys` для гибкости при работе со сторонними API).
