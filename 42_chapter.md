## Глава 42. Взаимодействие с Java SAM

### 1. Что такое SAM-интерфейс в Java?

В Java (до появления лямбд в версии 8) функции не были самостоятельными объектами. Если ты хотел передать кусок кода (поведение) в другой метод, тебе приходилось создавать объект, реализующий интерфейс.

Интерфейс, у которого есть **строго один абстрактный метод**, называется **SAM-интерфейсом**.
Классические примеры из Java: `Runnable` (метод `run()`), `Callable` (метод `call()`), `OnClickListener` (метод `onClick()`).

В старой Java передача клика по кнопке выглядела как огромный кусок шаблонного кода (анонимный класс):

```java
// --- СТАРЫЙ КОД НА JAVA ---
button.setOnClickListener(new OnClickListener() {
    @Override
    public void onClick(View v) {
        System.out.println("Кнопка нажата!");
    }
});

```

### 2. Магия автоматической конверсии (Из Kotlin в Java)

В Kotlin мы используем лямбды (функции первого класса). Лямбда имеет тип `() -> Unit` или `(View) -> Unit`. Но метод Java-библиотеки `setOnClickListener` ничего не знает про типы Kotlin. Он ждет объект с интерфейсом `OnClickListener`.

Как подружить эти два мира?
Компилятор Kotlin включает **автоматическую SAM-конверсию**. Если ты вызываешь метод Java, который ожидает SAM-интерфейс, Kotlin разрешает тебе просто передать туда лямбду!

* **Аналогия из жизни:** Бюрократия. Java-библиотека — это строгий чиновник, который принимает только официальные бланки по форме 4-Б (интерфейс). Ты (разработчик на Kotlin) ленив, ты просто пишешь текст на клочке бумаги (лямбда). Но у тебя есть гениальный помощник (компилятор Kotlin). Как только ты отдаешь клочок бумаги, помощник мгновенно переписывает твой текст на официальный бланк и отдает чиновнику. Чиновник доволен, а ты не тратил время на заполнение шапки документа.

```kotlin
// --- КОД НА KOTLIN ---

// Мы вызываем Java-метод и передаем ему обычную Kotlin-лямбду!
button.setOnClickListener { view -> 
    println("Кнопка нажата!")
}

// Что делает компилятор под капотом (он сам создает анонимный класс):
// button.setOnClickListener(object : OnClickListener {
//     override fun onClick(v: View) {
//         println("Кнопка нажата!")
//     }
// })

```

То же самое работает с любыми Java API, например, с пулами потоков (передача `Runnable`):

```kotlin
val executor = Executors.newSingleThreadExecutor()

// ExecutorService.submit() в Java ждет Runnable. 
// Мы просто кидаем ему лямбду:
executor.submit { 
    println("Работаю в фоновом потоке: ${Thread.currentThread().name}") 
}

```

### 3. Ловушка SAM-конверсии (Проблема отписки)

В этой магии есть один смертельно опасный нюанс, который часто приводит к утечкам памяти (Memory Leaks) в Android.

Как мы помним из аналогии, компилятор-помощник каждый раз **создает новый официальный бланк (новый объект интерфейса)**, когда ты передаешь лямбду.

Если тебе нужно не только подписаться на события, но и позже **отписаться** от них (удалить слушателя), передача лямбды напрямую сломает программу.

```kotlin
// ОПАСНЫЙ КОД!
fun bind(button: Button) {
    // 1. Компилятор создает НОВЫЙ объект OnClickListener_1
    button.setOnClickListener { println("Клик!") }
}

fun unbind(button: Button) {
    // 2. Компилятор создает НОВЫЙ объект OnClickListener_2
    button.removeOnClickListener { println("Клик!") } 
    // ОШИБКА ЛОГИКИ! Кнопка ничего не удалит, потому что OnClickListener_2 
    // никогда не был в списке слушателей. Мы потеряли ссылку на первый объект!
}

```

**Как правильно (Лечение):**
Если тебе нужно сохранить ссылку на созданный объект интерфейса, ты должен явно вызвать конструктор SAM-интерфейса. Это называется **Явная SAM-конверсия**.

```kotlin
// ПРАВИЛЬНЫЙ КОД
// Мы явно вызываем "конструктор" интерфейса, передавая ему лямбду.
// Теперь мы владеем реальным Java-объектом и можем сохранить его в переменную!
val myListener = OnClickListener { println("Клик!") }

fun bind(button: Button) {
    button.setOnClickListener(myListener) // Передаем объект
}

fun unbind(button: Button) {
    button.removeOnClickListener(myListener) // Отписываем ТОТ ЖЕ САМЫЙ объект
}

```

### 4. Функциональные интерфейсы внутри самого Kotlin (`fun interface`)

Долгие годы SAM-конверсия в Kotlin работала **только** для интерфейсов, написанных на Java. Если ты писал свой собственный интерфейс на Kotlin, компилятор отказывался конвертировать лямбды, заставляя тебя писать уродливые `object : MyInterface { ... }`.

Но начиная с Kotlin 1.4, разработчики добавили эту возможность в сам язык! Чтобы включить SAM-конверсию для своего Kotlin-интерфейса, нужно добавить перед ним слово **`fun`** (функциональный интерфейс).

```kotlin
// 1. Создаем интерфейс с одним методом на Kotlin. 
// Обязательно пишем 'fun' перед 'interface'!
fun interface Condition {
    fun check(number: Int): Boolean
}

// Функция, принимающая наш интерфейс
fun filterNumbers(list: List<Int>, condition: Condition): List<Int> {
    return list.filter { condition.check(it) }
}

fun main() {
    val numbers = listOf(1, 2, 3, 4, 5)

    // 2. МАГИЯ РАБОТАЕТ! 
    // Мы можем передавать лямбду, и она автоматически станет объектом Condition
    val evens = filterNumbers(numbers) { it % 2 == 0 }
    
    println(evens) // Выведет: [2, 4]
}

```

*(Важно: У `fun interface` может быть только один абстрактный метод. Но у него может быть сколько угодно обычных методов с готовой реализацией).*

---

Итог:

* **SAM (Single Abstract Method)** — это интерфейс, содержащий ровно один абстрактный метод. В Java они исторически использовались для передачи логики (колбэков) в функции.
* **Автоматическая SAM-конверсия** позволяет вызывать методы Java, ожидающие SAM-интерфейс, передавая им лаконичные Kotlin-лямбды. Компилятор сам оборачивает лямбду в анонимный объект нужного интерфейса на уровне байткода.
* **Риск утечек при отписке:** Поскольку автоматическая конверсия каждый раз создает *новый* инстанс объекта (экземпляр интерфейса), лямбды нельзя использовать напрямую для методов отписки (например, `removeListener`). Необходимо явно создать и сохранить объект в переменную (`val listener = OnClickListener { ... }`).
* **`fun interface`:** Специальный синтаксис, добавленный в Kotlin 1.4, который позволяет объявлять функциональные интерфейсы прямо в Kotlin-коде. Это включает для них ту же магию автоматической конверсии лямбд, которая ранее была доступна только для Java-интерфейсов.
