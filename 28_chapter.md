## Глава 28. Хвостовая рекурсия (tailrec)

### 1. Проблема обычной рекурсии: `StackOverflowError`

Рекурсия — это когда функция вызывает саму себя. Это очень красивый подход для обхода деревьев, парсинга или математических вычислений. Но в мире JVM (Java Virtual Machine) у нее есть фатальный недостаток — она потребляет память стека вызовов (Call Stack).

* **Аналогия из жизни:** Представь длинную цепочку бюрократов (субподрядчиков). Ты просишь Главного менеджера посчитать бюджет. Он не делает это сам, он вызывает Заместителя. Заместитель вызывает Бухгалтера. Бухгалтер вызывает Стажера.
**В обычной рекурсии** ни один из них не может уйти домой (освободить память), пока не получит ответ от нижестоящего. Менеджер стоит и ждет заместителя. Если цепочка состоит из 10 000 человек, в коридоре (в стеке вызовов) просто не хватит места, начнется давка, и программа упадет с ошибкой `StackOverflowError` (Переполнение стека).

Давай посмотрим на классическую, но опасную рекурсию — подсчет суммы чисел от N до 1:

```kotlin
// Обычная рекурсия. ОПАСНО ДЛЯ БОЛЬШИХ ЧИСЕЛ!
fun calculateSumBad(n: Long): Long {
    // Базовый случай: если дошли до 1, возвращаем 1 и начинаем сворачивать цепочку
    if (n == 1L) return 1L
    
    // ВНИМАНИЕ НА ЭТУ СТРОКУ:
    // Функция вызывает саму себя ( calculateSumBad(n - 1) ), 
    // НО после получения ответа она должна еще выполнить сложение ( n + ... ).
    // Из-за этого сложения функция вынуждена "висеть" в памяти и ждать ответа от следующей!
    return n + calculateSumBad(n - 1) 
}

fun main() {
    // Для 5 это сработает отлично (5 вызовов в памяти уместятся)
    println(calculateSumBad(5)) // Выведет 15

    // Но если мы попросим посчитать для 100 000:
    // calculateSumBad(100_000) // КРАШ ПРОГРАММЫ: java.lang.StackOverflowError
}

```

### 2. Хвостовая рекурсия (Tail Recursion) и паттерн `Accumulator`

Чтобы решить проблему, нужно сделать так, чтобы функции **не нужно было ничего ждать**. Вызов самой себя должен быть **самым последним (хвостовым) действием** во всем блоке кода. Никаких `+`, `*` или других операций после вызова быть не должно.

Для этого в программировании используют паттерн **Аккумулятор (Накопитель)**. Мы передаем промежуточный результат вперед, следующему звену.

* **Аналогия из жизни:** Сбор подписей по кабинетам. Ты заходишь к первому чиновнику с пустой папкой (аккумулятором). Он ставит свою подпись, кладет лист в папку и говорит: *"Я свою работу сделал, вот тебе папка, иди к следующему"*. Первый чиновник **сразу же уходит домой (освобождает память)**. Тебе не нужно к нему возвращаться. Всю информацию (текущую сумму) ты носишь с собой в папке от кабинета к кабинету.

Давай перепишем наш код с аккумулятором:

```kotlin
// Мы добавляем аргумент 'accumulator', который будет носить с собой текущую сумму.
// По умолчанию он равен 0, чтобы нам было удобно запускать функцию.
fun calculateSumBetter(n: Long, accumulator: Long = 0L): Long {
    // Если дошли до нуля, просто отдаем то, что накопили в "папке"
    if (n == 0L) return accumulator

    // САМОЕ ВАЖНОЕ: рекурсивный вызов - это АБСОЛЮТНО ПОСЛЕДНЕЕ действие.
    // Мы сразу вычисляем новую сумму (accumulator + n) и передаем её следующему вызову.
    // Нам не нужно ничего ждать!
    return calculateSumBetter(n - 1, accumulator + n)
}

```

Но погоди, в JVM даже такой идеальный код всё равно создаст новые вызовы в стеке и упадет! Виртуальная машина Java сама по себе не умеет оптимизировать хвостовую рекурсию.

Здесь в игру вступает магия компилятора Kotlin.

### 3. Ключевое слово `tailrec` (Оптимизация компилятора)

Чтобы компилятор Kotlin взял твой код и переписал его в быстрый, безопасный цикл, нужно просто добавить перед функцией слово **`tailrec`**.

```kotlin
// Добавляем слово tailrec (хвостовая рекурсия)
tailrec fun calculateSumPerfect(n: Long, accumulator: Long = 0L): Long {
    if (n == 0L) return accumulator
    return calculateSumPerfect(n - 1, accumulator + n)
}

fun main() {
    // ТЕПЕРЬ ЭТО АБСОЛЮТНО БЕЗОПАСНО!
    // Мы передаем огромное число, и программа работает мгновенно без переполнения памяти.
    val hugeSum = calculateSumPerfect(100_000L)
    println("Огромная сумма: $hugeSum") 
}

```

**Что происходит под капотом?**
Когда ты ставишь `tailrec`, компилятор Kotlin проверяет: действительно ли вызов функции является самым последним действием? Если да, то компилятор **вообще не использует рекурсию в байткоде**.

Он берет твою красивую рекурсивную функцию и физически переписывает её в обычный цикл `while`. Для процессора это выглядит примерно так:

```kotlin
// Вот во что компилятор Kotlin тайно превращает твой tailrec-код:
fun compiledSum(n: Long): Long {
    var currentN = n
    var currentAccumulator = 0L
    
    // Никаких вызовов функций и засорения памяти! Только быстрый цикл.
    while (currentN != 0L) {
        currentAccumulator = currentAccumulator + currentN
        currentN = currentN - 1
    }
    return currentAccumulator
}

```

*Если ты поставишь `tailrec`, но забудешься и напишешь `return n + func(...)` (сделаешь рекурсию не хвостовой), компилятор Kotlin заботливо подсветит слово `tailrec` желтым предупреждением и скажет: "Оптимизация невозможна, вызов не является хвостовым".*

---

Итог:

* Обычная рекурсия создает новый кадр (фрейм) в стеке вызовов для каждого повторения. При большом количестве итераций (тысячи вызовов) оперативная память переполняется, и программа аварийно завершается с ошибкой `StackOverflowError`.
* Рекурсия называется **хвостовой (tail call)**, если вызов самой себя является строго последней операцией в функции (после вызова не происходит никаких дополнительных вычислений, таких как сложение или умножение).
* Чтобы сделать рекурсию хвостовой, обычно используют дополнительный аргумент **Аккумулятор**, в котором хранится и передается на следующий шаг промежуточный результат.
* Ключевое слово **`tailrec`** в Kotlin указывает компилятору применить оптимизацию. Если правила хвостовой рекурсии соблюдены, компилятор на уровне байткода JVM заменяет рекурсивные вызовы на обычный, невероятно быстрый цикл (`while` / `goto`), полностью устраняя риск `StackOverflowError`.
