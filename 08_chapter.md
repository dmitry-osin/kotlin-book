## Глава 8. Иерархия: sealed классы и интерфейсы

### 1. Что такое `sealed` класс и зачем он нужен?

Слово `sealed` переводится как «запечатанный» (как конверт с сургучной печатью). Если ты объявляешь класс или интерфейс как `sealed`, ты говоришь компилятору: *"Все наследники этого класса находятся под моим строгим контролем. Никто извне (из других модулей или пакетов) не имеет права наследоваться от него"*.

Часто возникает вопрос: **А почему не использовать обычный `enum` (перечисление)?**
В `enum` все элементы абсолютно одинаковы по своей структуре. А `sealed` классы позволяют каждому состоянию иметь *свои собственные уникальные данные*.

* **Аналогия из жизни:** Способы оплаты на кассе магазина.
Магазин принимает строго три вида оплаты (это наш запечатанный список): Наличные, Банковская карта и Перевод по номеру телефона (СБП). Расплатиться пуговицами или бартером нельзя — иерархия закрыта.
При этом данные у каждого способа абсолютно разные:
* У **Наличных** есть сумма и номинал купюр, которые дал клиент.
* У **Карты** есть номер, срок действия и пин-код.
* У **СБП** есть только номер телефона и название банка.
Обычный `enum` не справится с таким разнообразием полей, а `sealed class` — легко.



```kotlin
// Создаем запечатанный класс. Создать объект самого PaymentMethod нельзя (он неявно abstract).
sealed class PaymentMethod 

// Наследник 1: Карта (хранит свои специфичные данные)
data class Card(val cardNumber: String, val pin: String) : PaymentMethod()

// Наследник 2: Наличные (хранит сумму и флаг, нужна ли сдача)
data class Cash(val amount: Double, val needsChange: Boolean) : PaymentMethod()

// Наследник 3: Перевод (хранит номер телефона)
data class MobileTransfer(val phoneNumber: String) : PaymentMethod()

// ОШИБКА! Если кто-то в другой части программы попытается добавить 
// класс Crypto : PaymentMethod(), компилятор это запретит. Иерархия запечатана!

```

### 2. Магия исчерпывающего `when` (Без ветки `else`)

Главная суперсила `sealed` классов раскрывается при работе с оператором `when`.

Так как компилятор знает **абсолютно всех** наследников запечатанного класса, он может математически гарантировать, что ты обработал все возможные варианты. Тебе больше не нужно писать ветку `else` "на всякий случай".

* **Аналогия из жизни:** Инструкция для кассира. Менеджер написал инструкцию: "Если дают карту — вставь в терминал. Если дают наличные — открой кассу. Если перевод — проверь пуш-уведомление". В этой инструкции нет пункта "Иначе (else) — делай что хочешь", потому что других способов оплаты в магазине просто не существует.

```kotlin
fun processPayment(method: PaymentMethod): String {
    // Оператор when проверяет тип объекта (с помощью ключевого слова 'is').
    // Компилятор автоматически делает умное приведение типов (Smart Cast).
    return when (method) {
        is Card -> "Оплата картой. Обработка номера: ${method.cardNumber}" 
        is Cash -> "Оплата наличными. Сумма: ${method.amount}. Нужна сдача: ${method.needsChange}"
        is MobileTransfer -> "Ожидаем перевод по номеру ${method.phoneNumber}"
        // Ветка 'else' ЗДЕСЬ НЕ НУЖНА И ДАЖЕ ВРЕДНА!
        // Компилятор видит, что мы перебрали все 3 возможных варианта из sealed class.
    }
}

```

### 3. Защита от багов в будущем

Почему отсутствие `else` — это так важно? Представь, что прошел год, и бизнес решил добавить оплату криптовалютой.

В старых языках программирования с обычным интерфейсом ты бы добавил новый класс `Crypto`. Но если бы в коде где-то стоял `when` с веткой `else`, оплата криптой тихо провалилась бы в `else` (например, выдав ошибку "Неизвестный метод"), и ты бы узнал об этом только от злых клиентов в продакшене.

В Kotlin с `sealed` классами всё иначе:

1. Ты открываешь файл и добавляешь: `data class Crypto(val wallet: String) : PaymentMethod()`
2. Ты нажимаешь "Собрать проект".
3. **Компилятор мгновенно бьет по рукам (выдает ошибку)** в функции `processPayment`. Он скажет: *"Эй! У тебя появился новый наследник `Crypto`, а в `when` для него нет правила! Код не скомпилируется, пока ты его не допишешь"*.

Это невероятно мощный архитектурный инструмент. Он переносит ответственность за проверку логики с разработчика (который может что-то забыть) на компилятор (который не забывает ничего).

---

Итог:

* `sealed class` (или `sealed interface`) создает строго ограниченную, запечатанную иерархию классов. Компилятор точно знает все классы, которые от него наследуются.
* В отличие от `enum`, где все элементы имеют одинаковую структуру, наследники `sealed` класса могут быть полноценными классами (`data class`, обычный `class`) со своим уникальным набором свойств.
* При использовании `sealed` классов в операторе `when` компилятор требует обработать все возможные варианты (исчерпывающая проверка), что позволяет безопасно избавиться от ветки `else`.
* Если в будущем в `sealed` иерархию добавится новый наследник, компилятор принудительно заставит разработчика обновить все конструкции `when`, где используется этот класс, полностью исключая риск забыть обработать новое состояние.
