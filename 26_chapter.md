## Глава 26. Делегирование

### 1. Делегирование реализации интерфейсов

Представь, что у тебя есть класс, который должен выполнять какой-то контракт (интерфейс), но ты не хочешь писать эту логику с нуля, потому что кто-то другой уже отлично с этим справляется. Ты можешь просто передать (делегировать) эти обязанности ему.

* **Аналогия из жизни:** Ресторан и шеф-повар. Менеджер ресторана юридически обязан предоставлять услугу «Приготовление еды» (реализует интерфейс `Cook`). Но сам менеджер не стоит у плиты. Он нанимает профессионального шеф-повара и **делегирует** эту обязанность ему. Когда клиент просит менеджера приготовить стейк, запрос автоматически перенаправляется на кухню к повару.

В Java менеджеру пришлось бы вручную писать метод `cook()`, внутри которого он бы вызывал `chef.cook()`. В Kotlin компилятор сделает это за нас.

```kotlin
// Наш контракт (интерфейс)
interface Cook {
    fun prepareDish(dishName: String)
}

// Профессионал, который реально умеет это делать
class ProfessionalChef(val name: String) : Cook {
    override fun prepareDish(dishName: String) {
        println("Шеф $name виртуозно готовит: $dishName")
    }
}

// Менеджер ресторана. Он ОБЯЗАН быть Cook (реализовывать интерфейс), 
// но он делегирует эту работу объекту chef с помощью ключевого слова 'by'.
class RestaurantManager(val chef: Cook) : Cook by chef {
    // Нам НЕ НУЖНО писать override fun prepareDish()!
    // Компилятор сам невидимо создаст этот метод и перенаправит вызов в объект chef.
    
    fun manage() {
        println("Я управляю рестораном, считаю деньги...")
    }
}

fun main() {
    // Нанимаем Гордона Рамзи
    val gordon = ProfessionalChef("Гордон Рамзи")
    
    // Создаем менеджера и передаем ему повара
    val manager = RestaurantManager(gordon)
    
    // Менеджер занимается своими делами
    manager.manage()
    
    // Клиент просит менеджера приготовить еду. 
    // Менеджер "сам" отвечает на вызов, но работу делает Гордон!
    manager.prepareDish("Говядина Веллингтон") 
}

```

Это невероятно мощный инструмент для создания классов-оберток (Decorator Pattern) с нулевым написанием шаблонного кода.

### 2. Делегирование свойств и Ленивая инициализация (`by lazy`)

Ключевое слово `by` можно применять не только к классам, но и к отдельным переменным (свойствам). Мы можем поручить логику получения значения (`get`) другому объекту-делегату.

Самый популярный и важный встроенный делегат в Kotlin — это **`lazy`** (ленивый).

Часто в программировании создание объекта требует много ресурсов: это может быть тяжелый запрос к базе данных, чтение огромного файла или сложный математический расчет. Если мы сделаем это прямо при старте программы, она будет долго загружаться.

* **Аналогия из жизни:** Покупка абонемента в спортзал. Ленивая инициализация — это когда ты покупаешь абонемент, но не собираешь спортивную сумку, не покупаешь форму и не нанимаешь тренера (тяжелая инициализация) в тот же день. Ты делаешь это **только в тот момент**, когда впервые реально решаешь пойти на тренировку. А когда ты пойдешь туда во второй, третий и сотый раз — форма уже собрана, ты просто берешь её (результат кешируется).

Делегат `by lazy` работает точно так же. Блок кода внутри него выполнится **строго один раз** при самом первом обращении к переменной. Все последующие обращения будут мгновенно получать уже готовый результат.

*(Важно: `lazy` работает только с неизменяемыми переменными `val`).*

```kotlin
class HeavyApplication {

    // Переменная не инициализируется при создании объекта HeavyApplication!
    // Мы поручаем её инициализацию делегату lazy.
    val databaseConnection: String by lazy {
        println("--> [СИСТЕМА]: Идет тяжелое подключение к базе данных (заняло 5 секунд)...")
        // Последняя строчка блока становится сохраненным значением
        "Подключение установлено: PostgreSQL"
    }

    fun start() {
        println("Приложение запущено. Интерфейс отрисован. База пока не тронута.")
    }
}

fun main() {
    val app = HeavyApplication()
    app.start() // Выведет только сообщение о запуске
    
    println("\nПользователь нажал кнопку 'Загрузить профиль'...")
    
    // ПЕРВОЕ обращение к переменной. 
    // Именно сейчас выполнится блок внутри lazy и напечатает лог "[СИСТЕМА]..."
    println("Результат 1: ${app.databaseConnection}") 
    
    println("\nПользователь снова нажал кнопку...")
    
    // ВТОРОЕ обращение. Блок кода больше НЕ ВЫПОЛНЯЕТСЯ. 
    // Мы мгновенно получаем закешированный результат. Никаких логов "[СИСТЕМА]..." не будет.
    println("Результат 2: ${app.databaseConnection}") 
}

```

### 3. Потокобезопасность `lazy` (Под капотом)

Для уровня эксперта важно понимать, как `lazy` ведет себя в высоконагруженных многопоточных системах.

Что если два потока (например, два пользователя) одновременно нажмут кнопку и попытаются обратиться к `lazy` переменной в самую первую миллисекунду? Не произойдет ли двойная инициализация базы данных?

По умолчанию делегат `lazy` в Kotlin **потокобезопасен (Thread-Safe)**.
Под капотом он использует механизм блокировок (synchronized). Если один поток начал инициализацию, все остальные потоки выстроятся в очередь и будут ждать, пока он не закончит. Как только первый поток запишет результат в память, остальные потоки просто считают уже готовое значение.

Тебе не нужно писать сложный код с блокировками (Double-Checked Locking), который так часто встречается в Java — Kotlin берет всю грязную работу с потоками на себя.

---

Итог:

* В Kotlin реализован нативный механизм делегирования через ключевое слово **`by`**, который позволяет элегантно применять архитектурный принцип "Композиция вместо наследования".
* **Делегирование интерфейсов (`Class : Interface by Object`)** позволяет классу объявить, что он реализует интерфейс, но физически перенаправить все вызовы его методов другому объекту. Компилятор автоматически генерирует весь необходимый шаблонный код для переадресации вызовов.
* **Делегирование свойств** позволяет передать логику чтения (и записи) переменной специальному объекту-делегату.
* **Ленивая инициализация (`val x by lazy { ... }`)** откладывает тяжелые вычисления до момента самого первого обращения к переменной. После первого выполнения результат надежно кешируется в оперативной памяти.
* По умолчанию делегат `by lazy` является потокобезопасным: он гарантирует, что даже в условиях многопоточности блок инициализации выполнится ровно один раз.
